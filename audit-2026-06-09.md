## Executive Summary

The backend is in materially better shape than the UI. The **server is genuinely multi-tenant-safe** (every school-scoped route derives `school_id` and `role` from the JWT-resolved DB row, never from the request body — see DOMAIN 8) and the **teacher vertical writes real data to Supabase with correct parent notifications** (DOMAIN 6). The biggest remaining problems are on the **client / cross-role-glue** layer:

- **🔴 Session state-bleed** — the singleton Ktor `HttpClient` `Auth` bearer cache is never cleared on logout (`clearToken()` does not exist anywhere). A logout → re-login as a different role can serve stale bearer tokens from cache.
- **🔴 Fabricated student data shown in production** — `PewsPreview` (admin home) and `AiReportCardPreview` (parent academics) render hardcoded fake students/grades ("Aman Verma 9-B 78", "180 students scored") as if they were live signals.
- **🟠 Parent login asks returning parents for their NAME** *(this audit's newest reported bug)* — the OTP step unconditionally shows a "Your name" field even on `LOGIN_PHONE`, where the ViewModel ignores it entirely.
- **🟠 Notification asymmetry** — teachers notify parents on every write (attendance/marks/homework/leave/message); **admins do not notify on direct messages**, and **admin results-publish sends no notification**.
- **🟠 Reply-only messaging on parent & admin sides** — the API supports compose-new (`recipient_user_id`) but neither the parent nor the admin UI exposes a "new message / pick recipient" affordance.
- **🟡 Parent multi-child non-propagation** — each parent ViewModel holds its own `selectedChildId`; switching child on one tab does not carry to the others.
- **🟡 Cosmetic hardcoding** — hardcoded "Parent" avatar name, always-on red bell dots, a fixed Comms-tab badge of `2`, and fabricated landing social-proof ("240+ schools · 38k parents").

**Second deliverable status (premium common landing page):** **Already substantially built.** `CommonLandingScreenV2.kt` is a full premium landing surface in the V* design system, carries the legacy content forward, exposes **both** Parent and Admin login entries, and already uses the 3 bundled campus images. Only the fabricated social-proof line needs honesty cleanup. See DOMAIN 2.

### Addendum — four user-reported issues audited 2026-06-09 (RA-S17 … RA-S20)

A second round of user reports was triaged against the live code. All four are **confirmed** and added below; none change the security posture (RA-S14 still holds).

- **🟠 RA-S17 (People tab) — architecture gap, not a bug.** The admin People tab is a flat scroll with a Teacher roster + Student roster card and **a direct "Remove" button on each teacher row**. There are **no sub-tabs** (Teachers / Students / Non-teaching staff), **no non-teaching-staff feature at all** (no table, no route, no UI), **no search/filter on the tab**, and **deletion is not inside the profile** — the profiles (`StudentProfileScreenV2`, `TeacherProfileScreenV2`) have no delete affordance. User's explicit ask: sub-tabs + clickable real DB-backed profiles + search/filter + move delete (with confirm popup) into the profile and **remove the direct row button**.
- **🟠 RA-S18 (Teacher attendance "Submit" persists) — ROOT CAUSE FOUND; it is a UI bug, NOT a DB bug.** Attendance **does** persist: `TeacherRoutingTasks.kt` POST `/attendance` upserts `attendance_records` and notifies parents (verified, same path as RA-S15). So the button fakes a success animation and reverts to "Submit" regardless of the real outcome.
- **🟠 RA-S19 (Parents can't see attendance/marks) — data plane is wired & correct; the gap is the attendance UX + a fragile marks join.** The child-scoped read endpoints exist and are correctly joined (`/parent/child/{id}/{attendance,marks,syllabus}`, ownership-gated, registered in `Application.kt`); teacher write keys match parent read keys (`person_id == student_code`). RA-S11 (fabricated `AiReportCardPreview` on the same screen) compounds the confusion.
- **🟡 RA-S20 (Common landing worse than demanded).** Screenshots confirm `CommonLandingScreenV2` renders but reads plain/generic versus the premium bar, and still shows the **fabricated** "Bank-grade encryption" pill + "240+ schools · 38k parents" strip (this is RA-S13's copy). Supersedes the over-optimistic RA-S16 "DONE": the surface exists but needs an honesty pass + visual polish, not a rebuild.

# DOMAIN 1 — Session & Authentication

### RA-S01 · 🔴 CRITICAL · ✅ FIXED `15ede8c` · Bearer-token cache is never cleared on logout (session state-bleed)
- **Location:** `shared/src/commonMain/kotlin/com/littlebridge/vidyaprayag/di/Koin.kt` (HttpClient `single { … Auth { bearer { loadTokens/refreshTokens } } }`, ~lines 60–112); `shared/.../feature/auth/data/repository/AuthRepositoryImpl.kt` (`logout()` → `clearSession()`).
- **Problem:** The `HttpClient` is a Koin `single` (one instance for the process). It installs the Ktor `Auth` plugin with a `bearer { loadTokens { … }; refreshTokens { … } }` block. The Ktor `Auth`/`BearerAuthProvider` keeps an **in-memory token cache**. On logout the app calls `clearSession()` (clears DataStore) — but **never calls `Auth` provider `clearToken()`**. A repo-wide grep for `clearToken` returns **zero** hits.
- **Evidence:**
  ```
  $ grep -rn "clearToken" shared/src/commonMain/...   → (no results)
  ```
  `PreferenceRepository.clearSession()` removes `USER_ROLE / TOKEN / ID / REFRESH / PROFILE_COMPLETED` only. The bearer provider's cached `BearerTokens` survive because the singleton client is never recreated and `clearToken()` is never invoked.
- **Impact:** After logout → re-login (especially as a **different role** on the same device), the first authenticated requests can still be signed with the **previous user's cached bearer token** until a 401 forces a refresh. Cross-role data exposure and "ghost session" bugs. This is the root of the recurring "logged out but still sees old data" class of issues.
- **Fix spec:**
  1. Expose a way to clear the provider: hold a reference to the `BearerAuthProvider` (or `client.plugin(Auth).providers`) and call `.clearToken()` on logout.
  2. In `AuthRepositoryImpl.logout()` (and any forced-logout path), after `clearSession()`, invoke the bearer clear. Minimal pattern:
     ```kotlin
     // in DI, after building client:
     client.plugin(Auth).providers
         .filterIsInstance<BearerAuthProvider>()
         .forEach { it.clearToken() }
     ```
     Wrap this in a small `SessionManager.clearAuthCache()` that `logout()` calls.
  3. Belt-and-braces: `loadTokens` must read the **current** DataStore values each call (verify it isn't capturing a stale snapshot); return `null` when no token is stored so a logged-out client never sends a token.

### RA-S02 · 🟠 HIGH · ✅ FIXED · Returning parent is asked for their NAME on the OTP step *(newly reported in this audit)*
- **Fix:** `ParentAuthScreenV2.kt` `AuthStep.Otp` branch now renders the "Your name" field **only when `state.flow == AuthFlow.SIGNUP_PHONE`** (new family). A returning parent (`LOGIN_PHONE`) sees only the OTP field. UI-only change — the `AuthViewModel.onSubmit()` already gated the name *requirement* and the `login()` payload on `SIGNUP_PHONE`, so this aligns the UI with the existing VM contract (no name collected/sent on login).
- **Location:** `composeApp/src/commonMain/kotlin/com/littlebridge/vidyaprayag/ui/v2/screens/auth/ParentAuthScreenV2.kt` — `AuthStep.Otp` branch.
- **Problem:** A parent enters their phone number; the server's `checkUser` decides the flow. Both **`LOGIN_PHONE`** (existing account) and **`SIGNUP_PHONE`** (new account) land on the same UI step, `AuthStep.Otp` (confirmed in `AuthViewModel.onContinue()`, line 67: `AuthFlow.LOGIN_PHONE, AuthFlow.SIGNUP_PHONE -> AuthStep.Otp`). The screen then **unconditionally** renders a "Your name" `VInput` above the OTP field — even for a returning user logging in.
- **Evidence:**
  ```kotlin
  // ParentAuthScreenV2.kt — AuthStep.Otp branch
  AuthStep.Otp -> {
      // Phone signup needs a name alongside the OTP; show it inline for new families.
      // (For an existing-user login the field stays empty and is ignored by the VM.)
      VInput(
          value = state.name,
          onValueChange = viewModel::onNameChanged,
          label = "Your name",                // ← shown to RETURNING users too
          placeholder = "Full name",
          leadingIcon = VIcons.User,
          modifier = Modifier.fillMaxWidth(),
      )
      Spacer(Modifier.height(16.dp))
      VInput(value = state.otp, ... label = "OTP" ...)
  }
  ```
  The ViewModel **already** does the right thing — it only requires/sends `name` when the flow is signup (`AuthViewModel.onSubmit()`, lines 125–145):
  ```kotlin
  AuthStep.Otp -> {
      if (currentState.flow == AuthFlow.SIGNUP_PHONE && currentState.name.isBlank()) { … "Please enter your name" … }
      if (currentState.flow == AuthFlow.SIGNUP_PHONE) {
          repository.signup(SignupRequest(name = currentState.name, …))
      } else {
          repository.login(LoginRequest(identifier = …, otp = …, role = …))   // name NOT sent
      }
  }
  ```
  So the field is **purely UI cruft for returning users** — collected, displayed, then ignored. The user's complaint is exactly right: *"WHEN I'M LOGGING AS PARENT WHOSE ACCOUNT HAS ALREADY [BEEN] CREATED IT STILL ASKS FOR NAME AND OTP — WHY ASK FOR NAME WHEN [IT WAS] CREATED ALREADY? JUST ASK FOR OTP."*
- **Impact:** Returning parents are confronted with a "Your name" box on every login. It looks like the app forgot who they are; it invites them to type a name that does nothing; it erodes trust in a sign-in flow that should feel instant. New families still need the field — so the fix must be flow-aware, not a blanket removal.
- **Fix spec (UI-only, no VM change needed):** Gate the name field on the signup flow inside the `AuthStep.Otp` branch:
  ```kotlin
  import com.littlebridge.vidyaprayag.feature.auth.domain.model.AuthFlow
  // …
  AuthStep.Otp -> {
      if (state.flow == AuthFlow.SIGNUP_PHONE) {        // ← only new families see the name field
          VInput(
              value = state.name,
              onValueChange = viewModel::onNameChanged,
              label = "Your name",
              placeholder = "Full name",
              leadingIcon = VIcons.User,
              modifier = Modifier.fillMaxWidth(),
          )
          Spacer(Modifier.height(16.dp))
      }
      VInput(value = state.otp, onValueChange = viewModel::onOtpChanged, label = "OTP", …)
      Spacer(Modifier.height(8.dp))
      Text("We sent a code to ${state.identifier.ifBlank { "your phone" }}.", …)
  }
  ```
  - `state.flow` is available on `AuthUiState.flow: AuthFlow?` and is set by `onContinue()` before the step changes.
  - Optionally tighten the subtitle copy so returning users read "Enter the code we just sent you" while new families read "Create your family account."
  - **Acceptance:** existing parent → enters phone → sees **only** the OTP field; new parent → enters phone → sees name + OTP. No VM changes required.

### RA-S05 · 🟡 MEDIUM · ✅ FIXED · Multi-child `selectedChildId` is not propagated across parent ViewModels

**Fix (server `8350888` + shared `9d85d01`):** New `core/state/SelectedChildHolder` (Koin `single` exposing `StateFlow<String?>`, with `select` / `selectIfUnset` / `clear`) is the single source of truth for the parent's active child. `ParentHomeViewModel`, `ParentAcademicsViewModel`, and `ParentLeaveViewModel` now **observe** the holder (a child picked on one tab updates the others) and **publish** their pick via `selectChild`; on load they seed the holder with the default first child via `selectIfUnset` (without clobbering an explicit choice). `FeeViewModel` observes the holder and re-fetches fees scoped to the active child — `getFees(token, childId?)` was threaded through `ParentApi` / `ParentRepository` / `ParentRepositoryImpl`, and the server `GET /api/v1/parent/fees` gained an optional `?child_id=` filter (parent-scoped fetch, then in-memory child filter — Postgres-portable). `AuthRepositoryImpl.logout()` calls `selectedChildHolder.clear()` so a re-login as a different parent starts fresh (the single survives logout). `childId` defaults to `null` everywhere → backward-compatible (all-records view when nothing is selected).
- **Location:** `shared/.../feature/parent/presentation/{ParentHomeViewModel, ParentAcademicsViewModel, FeeViewModel, ParentMessageViewModel, ParentProfileViewModel}.kt`.
- **Problem:** Each parent ViewModel owns its **own** `selectedChildId` with no shared source of truth. `FeeViewModel.getFees(token)` takes no `childId` at all. Switching the active child on one tab does not change the others.
- **Impact:** A parent with two children who selects "Child B" on the Home tab still sees "Child A" data on Academics/Fees. Confusing and potentially shows the wrong child's marks/fees.
- **Fix spec:** Introduce a single shared `SelectedChildHolder` (a `single` in Koin exposing `StateFlow<String?>`); each parent ViewModel observes it instead of holding its own. Thread `childId` into `FeeViewModel.getFees(token, childId)` and the corresponding `/parent/fees?child_id=` query.
