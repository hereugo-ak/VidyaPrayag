## Executive Summary

The backend is in materially better shape than the UI. The **server is genuinely multi-tenant-safe** (every school-scoped route derives `school_id` and `role` from the JWT-resolved DB row, never from the request body ŌĆö see DOMAIN 8) and the **teacher vertical writes real data to Supabase with correct parent notifications** (DOMAIN 6). The biggest remaining problems are on the **client / cross-role-glue** layer:

- **­¤ö┤ Session state-bleed** ŌĆö the singleton Ktor `HttpClient` `Auth` bearer cache is never cleared on logout (`clearToken()` does not exist anywhere). A logout ŌåÆ re-login as a different role can serve stale bearer tokens from cache.
- **­¤ö┤ Fabricated student data shown in production** ŌĆö `PewsPreview` (admin home) and `AiReportCardPreview` (parent academics) render hardcoded fake students/grades ("Aman Verma 9-B 78", "180 students scored") as if they were live signals.
- **­¤¤Ā Parent login asks returning parents for their NAME** *(this audit's newest reported bug)* ŌĆö the OTP step unconditionally shows a "Your name" field even on `LOGIN_PHONE`, where the ViewModel ignores it entirely.
- **­¤¤Ā Notification asymmetry** ŌĆö teachers notify parents on every write (attendance/marks/homework/leave/message); **admins do not notify on direct messages**, and **admin results-publish sends no notification**.
- **­¤¤Ā Reply-only messaging on parent & admin sides** ŌĆö the API supports compose-new (`recipient_user_id`) but neither the parent nor the admin UI exposes a "new message / pick recipient" affordance.
- **­¤¤Ī Parent multi-child non-propagation** ŌĆö each parent ViewModel holds its own `selectedChildId`; switching child on one tab does not carry to the others.
- **­¤¤Ī Cosmetic hardcoding** ŌĆö hardcoded "Parent" avatar name, always-on red bell dots, a fixed Comms-tab badge of `2`, and fabricated landing social-proof ("240+ schools ┬Ę 38k parents").

**Second deliverable status (premium common landing page):** **Already substantially built.** `CommonLandingScreenV2.kt` is a full premium landing surface in the V* design system, carries the legacy content forward, exposes **both** Parent and Admin login entries, and already uses the 3 bundled campus images. # DOMAIN 1 ŌĆö Session & Authentication

### RA-S01 ┬Ę ­¤ö┤ CRITICAL ┬Ę Ō£ģ FIXED `15ede8c` ┬Ę Bearer-token cache is never cleared on logout (session state-bleed)
- **Location:** `shared/src/commonMain/kotlin/com/littlebridge/vidyaprayag/di/Koin.kt` (HttpClient `single { ŌĆ” Auth { bearer { loadTokens/refreshTokens } } }`, ~lines 60ŌĆō112); `shared/.../feature/auth/data/repository/AuthRepositoryImpl.kt` (`logout()` ŌåÆ `clearSession()`).
- **Problem:** The `HttpClient` is a Koin `single` (one instance for the process). It installs the Ktor `Auth` plugin with a `bearer { loadTokens { ŌĆ” }; refreshTokens { ŌĆ” } }` block. The Ktor `Auth`/`BearerAuthProvider` keeps an **in-memory token cache**. On logout the app calls `clearSession()` (clears DataStore) ŌĆö but **never calls `Auth` provider `clearToken()`**. A repo-wide grep for `clearToken` returns **zero** hits.
- **Evidence:**
  ```
  $ grep -rn "clearToken" shared/src/commonMain/...   ŌåÆ (no results)
  ```
  `PreferenceRepository.clearSession()` removes `USER_ROLE / TOKEN / ID / REFRESH / PROFILE_COMPLETED` only. The bearer provider's cached `BearerTokens` survive because the singleton client is never recreated and `clearToken()` is never invoked.
- **Impact:** After logout ŌåÆ re-login (especially as a **different role** on the same device), the first authenticated requests can still be signed with the **previous user's cached bearer token** until a 401 forces a refresh. Cross-role data exposure and "ghost session" bugs. This is the root of the recurring "logged out but still sees old data" class of issues.
- **Fix spec:**
  1. Expose a way to clear the provider: hold a reference to the `BearerAuthProvider` (or `client.plugin(Auth).providers`) and call `.clearToken()` on logout.
  2. In `AuthRepositoryImpl.logout()` (and any forced-logout path), after `clearSession()`, invoke the bearer clear. Minimal pattern:
     ```kotlin
     // in DI, after building client:
     client.plugin(Auth).providers
         .filterIsInstance<BearerAuthProvider>()
         .forEach { it.clearToken() }
     ```
     Wrap this in a small `SessionManager.clearAuthCache()` that `logout()` calls.
  3. Belt-and-braces: `loadTokens` must read the **current** DataStore values each call (verify it isn't capturing a stale snapshot); return `null` when no token is stored so a logged-out client never sends a token.

### RA-S02 ┬Ę ­¤¤Ā HIGH ┬Ę Ō£ģ FIXED ┬Ę Returning parent is asked for their NAME on the OTP step *(newly reported in this audit)*
- **Fix:** `ParentAuthScreenV2.kt` `AuthStep.Otp` branch now renders the "Your name" field **only when `state.flow == AuthFlow.SIGNUP_PHONE`** (new family).   - Optionally tighten the subtitle copy so returning users read "Enter the code we just sent you" while new families read "Create your family account."
  - **Acceptance:** existing parent ŌåÆ enters phone ŌåÆ sees **only** the OTP field; new parent ŌåÆ enters phone ŌåÆ sees name + OTP. No VM changes required.

# DOMAIN 4 ŌĆö Communication System

### RA-S07 ┬Ę ­¤¤Ā HIGH ┬Ę Parent & admin messaging is reply-only (no compose-new UI) despite API support
- **Location (UI):** `composeApp/.../ui/v2/screens/parent/ParentMessagesScreenV2.kt` (reply-only); `composeApp/.../ui/v2/screens/school/MessagesScreenV2.kt` (line 95 wires `onSend = viewModel::sendReply` only). **(API):** `server/.../feature/school/MessagesRouting.kt` `POST /messages` accepts `recipient_user_id` (line 76, 252); the admin VM (`shared/.../feature/admin/presentation/MessagesViewModel.kt`) exposes both `sendMessage(recipientUserId)` and `sendReply`.
- **Problem:** The two-row conversation engine (`MessagingCore.sendInConversation`) fully supports **starting a new 1:1 thread** by passing `recipient_user_id`. But:
  - The **admin** screen only ever calls `sendReply` ŌĆö there is **no "New message" button, no recipient picker** in `MessagesScreenV2`. `sendMessage(recipientUserId)` is dead UI-wise.
