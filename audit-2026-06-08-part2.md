## 0. EXECUTIVE SUMMARY — what actually happens if you ship today

The **server multi-tenancy core is sound** and the **admin surface is the most complete**. But this paranoid re-read confirms the previous audit's three-spine failures **and surfaces a new top-of-stack security hole the previous audit missed entirely**:

### The single most important new finding

> **RA-53 🔴 — Anyone can self-register as a `school_admin` (and create a tenant) with zero verification, straight from the UI.** The landing screen's "School Administration" button opens `AdminAuthScreenV2`, which has a full **"Create account"** signup branch (`AdminAuthScreenV2.kt:118-166,185`). It calls `signup(role="ADMIN")` (`AuthViewModel.kt:118-123` after `onRoleChanged("ADMIN")` at `AdminAuthScreenV2.kt:63`). The public `/signup` route normalises `"ADMIN" → "school_admin"` and `"super_admin" → "super_admin"` (`AuthRouting.kt`, `roleNormalised`). On first login the onboarding wizard (`OnboardingRouting.kt:ensureSchoolForUser`) **auto-creates a `schools` row and promotes the user to `school_admin`**. Net effect: any anonymous person creates a school tenant and becomes its admin. This is wired end-to-end through the UI — not theoretical.

## AUDIT SECTION 2 — SECURITY & AUTH (re-verified + new)

### 2.1 RA-53 🔴 — Public self-service privilege escalation
**Path (end-to-end, all citations):**
1. `CommonLandingScreenV2` → `onAdmin` → `NavGraphV2.kt:131` routes to `AdminAuthScreenV2`.
2. `AdminAuthScreenV2.kt:63` `LaunchedEffect { viewModel.onRoleChanged("ADMIN") }`.
3. New email → `checkUser` returns `SIGNUP_EMAIL` → `AuthStep.SignupDetails` (`AuthViewModel.kt:64-68`).
4. `AdminAuthScreenV2.kt:118-166` renders name + password + confirm; CTA "Create account" (`:185`) → `viewModel.onSubmit()`.
5. `AuthViewModel.kt:118-123` → `repository.signup(SignupRequest(role="ADMIN"))`.
6. Server `/signup` (`AuthRouting.kt`) `roleNormalised`: `"admin"/"school_admin" → "school_admin"`, `"super_admin" → "super_admin"`. New `app_users` row, `profile_completed=false`, `school_id=null`.
7. First login → `NavGraphV2.AuthedFlow` sends `school_admin` with `profileCompleted==false` to `SchoolOnboardingScreenV2` → `OnboardingRouting.submit` → `ensureSchoolForUser` creates a `schools` row and the user is now a fully-provisioned school admin.

**Impact:** anonymous tenant creation; anyone becomes a `school_admin`. `super_admin` is also selectable via the same normaliser (no UI screen sets it today, but the route accepts it).
**Fix:** the public `/signup` must **only** ever mint `parent` (and only via the parent OTP path). Admin/teacher accounts must be **provisioned** (admins by an operator/allow-list or invite token; teachers already are, via `TeacherProvisioningRouting`). Concretely: in `/signup`, force `role = "parent"` regardless of body; remove the `SignupDetails` branch from `AdminAuthScreenV2` (admins log in, never self-sign-up); gate any admin creation behind a server-side invite/allow-list.

### 2.2 RA-54 🔴 — Teacher onboarding loop + no change-password
**Facts:**
- `TeacherProvisioningRouting.kt:148` creates teachers with `profileCompleted = false`.
- `profile_completed` is flipped to `true` in exactly three places: `DemoSeed.kt:116` (demo admin), `OnboardingRouting.kt:569` (school-admin completes onboarding), `ParentLinkRouting.kt:199` (parent links child). **No path flips a teacher's flag.**
- `NavGraphV2.kt:192` routes `Teacher` + `profileCompleted==false` → `TeacherFirstLoginScreenV2`. The NavGraph comment (`:170`) explicitly says this gate is **local-only** "until the documented `POST /auth/change-password` + `must_change_password` backend lands."
- `grep change-password` across `server/` and `shared/` = **0 hits** — the endpoint does not exist.

**Impact:** A provisioned teacher's `profile_completed` stays `false` forever. On every cold start the session reports `profileCompleted=false` → the first-login gate reappears (or is dismissed only in memory). The "force change initial password" promise is cosmetic; the generated password (`TeacherProvisioningRouting.generateInitialPassword`) can never be changed by the teacher.
**Fix:** add `POST /api/v1/auth/change-password` (verifies old/sets new via `PasswordHasher`, sets `app_users.profile_completed = true`, revokes other sessions); add `must_change_password boolean` (or reuse `profile_completed`) so the gate has a real server signal; the teacher client calls it and the gate resolves permanently.

### 2.3 Auth subsystem — verified sound (no change)
- **OTP** (`OtpService.kt`, `OtpAdminRouting.kt`, `delivery/*`): SHA-256 salt+pepper, `FOR UPDATE` row lock (RA-38), 10-min TTL, 5-attempt lock, 5-resend/hr cap, dev-return-code hard-gated to non-prod (RA-33). Multi-provider dispatch chain (WhatsApp/Fast2SMS/MSG91/Twilio/SMTP/Console) with env-driven ordering; `OtpAdminRouting` gated by `OTP_ADMIN_TOKEN` (404 when unset), constant-time compare. **Robust.**
- **Passwords** (`PasswordHasher.kt`): PBKDF2-HMAC-SHA256, 120k iterations, 16-byte salt, constant-time verify, legacy-hash compatibility + `needsRehash`. **Sound.**
- **Login throttle** (`LoginThrottle.kt`): sliding-window per-IP + per-identifier, success clears identifier window. In-memory (single-instance only — documented). **Sound.**
- **Error handling** (`ErrorHandling.kt`): StatusPages → JSON envelopes; SLF4J structured logging; `DEBUG_ERRORS` hard-gated to non-prod; 401 hint on `/api/v1/*` without bearer. **Sound.**
- **JWT** (`SecurityModule.kt`): HMAC256; `validate{}` re-checks `is_active` per request (RA-34) — see RA-62 for the perf note.
- **Storage** (`SupabaseStorage.kt`): service-role REST wrapper; object path always `{schoolId}/{kind}/{uuid}.{ext}` → tenant-isolated; `isConfigured()` → clean 503; type allow-list. **Sound.**

### 2.4 Carried open: RA-03 🟡 — roll-number enumeration
`ParentLinkRouting.link-child` resolves a canonical student by `(school_id, roll_number)` with no per-parent rate limit. A parent who has chosen a school can probe roll numbers. Mitigate with a throttle + the RA-48 admin-approval workflow.
