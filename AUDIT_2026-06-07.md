## 1. рџ”ґ DATABASE / SUPABASE SCHEMA LAYER

### 1.1 Two conflicting "source of truth" schemas
- **Root `/supabase_schema`** (Postgres v2.1) defines a *Supabase-native* model: `school_directory, schools, users (PK = auth.users.id, full RLS via auth.uid()/my_role()/my_school_id()), students (~30 cols), student_parent_link, daily_progress, academic_records, achievements, ai_reports, fee_structures, fee_payments`.
- **`docs/db/vidyasetu_schema.sql`** (CRLF line endings) defines the model the **server actually maps to**: `app_users, auth_otps, children, fee_records, exam_results, message_threads, messages, leave_requests, ptm_events, ptm_class_progress, users, schools (NO lat/long), students (thin 9-col)`.
- **The server (`server/.../db/Tables.kt`) maps to the SECOND schema, NOT the root one.** Yet `Tables.kt` lines 1вЂ“30 **falsely claim** it maps to `/supabase_schema` + `01_supplementary_schema.sql`. The header documentation is wrong.
- **Net:** `/supabase_schema` only creates **2 of the 36 server tables** (`schools`, `students`) and even those differ in column shape (root `students` ~30 cols vs server `StudentsTable` thin 9-col mirror at `Tables.kt:406`). Following the root schema alone leaves the server with 34 missing tables.

### 1.2 Definitive table-coverage matrix (the user's explicit "do we have all the DB tables?" ask)
`server/.../db/DatabaseFactory.kt` `allTables` (lines 44вЂ“83) registers **36 Exposed tables**. Postgres does NOT auto-create unless `AUTO_CREATE_TABLES=true`; otherwise you must run SQL by hand. There are two *documented* provisioning "recipes," and **both are incomplete**:

- **Recipe A** вЂ” what `Tables.kt`/`DatabaseFactory.kt` instruct (`supabase_schema` + `docs/backend/sql/01_supplementary_schema.sql`):
  **MISSING 15 tables:** `assessments, assessment_marks, syllabus_units, homework, homework_submissions, teacher_periods, teacher_subject_assignments, children, fee_records, exam_results, message_threads, messages, leave_requests, ptm_events, ptm_class_progress`.
- **Recipe B** вЂ” what `docs/db/migration_002_*.sql` run-order block (lines 22вЂ“24) instructs (`vidyasetu_schema.sql` в†’ `migration_001` в†’ `migration_002`):
  **MISSING 6 teacher tables:** `assessments, assessment_marks, syllabus_units, homework, homework_submissions, teacher_periods` (it omits `docs/backend/sql/02_teacher_schema.sql` entirely).
- **The ONLY complete recipe** is `vidyasetu_schema.sql` + `migration_001` + `migration_002` + `02_teacher_schema.sql` вЂ” and **no single document lists all four together.** A new dev will guess one of the two incomplete recipes and get 500s on the teacher/messaging/ptm/fee endpoints.

### 1.3 Column drift that will bite at runtime
- `SchoolsTable.latitude`/`longitude` (`Tables.kt:184вЂ“185`) exist in the server model and are used by the Haversine distance sort in `ParentDashboardRouting.kt:97`, but they are **NOT** in `vidyasetu_schema.sql` (`schools` at ~line 528) nor in the root schema вЂ” they are only added by `migration_002` (lines 45вЂ“48). Skip that migration в†’ `schools/discover` distance query throws "column does not exist."
- `FeeRecordsTable.currency` defaults to `"USD"` (`Tables.kt:464`) вЂ” wrong default for an India-first product (should be `INR`).
- FK columns across `Tables.kt` are plain `uuid()` with **no `.references()`** в†’ no referential integrity; orphaned `children`/`fee_records` rows are possible and silent.

## 3. рџ”ґ AUTHENTICATION / SESSION / ROLE LAYER

### 3.1 Teachers can never be created (portal unreachable)
- `AuthRouting.kt` signup (`~348`) writes `app_users` and login (`~395вЂ“475`) reads role from DB. `roleNormalised` (lines 173вЂ“180) yields `parent / school_admin / teacher / super_admin`.
- **There is no code path anywhere that creates a `teacher` row.** No teacher self-signup, no school-admin "add teacher" endpoint that provisions an `app_users` record with role `teacher`. So `login` can never find a teacher в†’ the whole `/api/v1/teacher/*` surface (`TeacherRouting.kt`, `TeacherRoutingTasks.kt`) and the teacher portal screens are **dead in production**.

### 3.2 Forgeable JWT secret
- `JwtConfig.kt:40` вЂ” `val secret by lazy { env("JWT_SECRET", "vidyaprayag-dev-secret-change-me") }`. If `JWT_SECRET` is unset in the Render env, the server signs with a **public, hardcoded string** в†’ anyone can forge an admin token. The header (lines 26вЂ“27) even admits it's "INSECURE." Must hard-fail at boot if unset in prod.
- Access-token TTL is **7 days** (`JwtConfig.kt:44`, `604800`s) and refresh is **30 days** (line 75) вЂ” long-lived tokens magnify the forgery risk.

### 3.3 Unsalted password hashing
- `AuthRouting.kt:165` вЂ” `hashPassword(p) = sha256Hex("pwd:$p")`. A single **static prefix, no per-user salt, no KDF (bcrypt/argon2)**. The whole password column is rainbow-table / GPU-crackable. Used at signup (line 348) and login compare (line 402).

### 3.4 Session persistence is lossy; client refresh is dead code
- Server **does** issue refresh tokens and persist `UserSessionsTable` (`AuthRouting.kt:362, 424, 504вЂ“506`) and exposes a working `/refresh` route validating `refresh_token_hash` (line 455).
- **But the client never uses any of it:**
  - `AuthApi.kt` exposes `checkUser/signup/login/sendOtp/verifyOtp/getUserDetails` вЂ” **no `refresh()` method**. The server's refresh infrastructure is therefore **unreachable / dead code**.
  - `AuthRepositoryImpl.saveSession()` (lines 73вЂ“77) persists **only** `token` + `role` to prefs and stashes the full `AuthResponse` in an **in-memory `cachedSession`**. `getSession()` returns only `cachedSession` в†’ **`null` after app restart.**
  - `PreferenceRepository` (`core/prefs/PreferenceRepository.kt`) has fields for **only** `themeName`, `userRole`, `userToken` вЂ” **no `profileCompleted`, no `userId`, no `refreshToken`.** These three are **permanently unrecoverable** after a restart, and the refresh token is never stored anywhere, so even adding a client `refresh()` wouldn't have a token to send.
- `Koin.kt` HttpClient (lines 59вЂ“73) installs **only** `ContentNegotiation`, `HttpRedirect`, `HttpTimeout`. **No `install(Auth)`, no 401 interceptor** в†’ an expired token surfaces as a generic error, never an auto-refresh.
- Silver lining (and a subtle inconsistency): most VMs read the token via `preferenceRepository.getUserToken()` (persisted), e.g. `FeeViewModel.kt:48`, so token-only calls survive restart вЂ” but anything depending on `getSession()` (e.g. `profileCompleted` gating) breaks.

### 3.5 `super_admin` mis-routes to Parent portal
- `NavGraphV2.kt` `EntryRole.from` (lines 80вЂ“85) maps `PARENT / ADMIN / SCHOOL_ADMIN / TEACHER` but **not `super_admin`** в†’ it falls to `Unknown` в†’ routed to `ParentPortal` (line 231). A super-admin lands in a parent UI with no admin surface.
- Same file: `AuthedFlow` reads `getSession()?.profileCompleted ?: true` (lines 182вЂ“183) вЂ” because `getSession()` is null after restart (В§3.4), this **defaults to `true`** and skips the profile-completion gate entirely on a cold start.

### 3.6 No server-side logout / revocation
- `AuthRepositoryImpl.logout()` clears prefs only вЂ” it does **not** call the server to revoke the `UserSessionsTable` row. Refresh tokens remain valid server-side for 30 days after "logout."

## 4. рџ”ґ SERIALIZATION PARITY вЂ” client DTO в†” server DTO (per-endpoint)

Method: compared every `@SerialName`/field of the client `domain/model` DTO against the server `*Routing.kt` `@Serializable` DTO, plus the envelope (`ApiResponse{success, message, data}` from `core/ResponseExtensions.kt`).

### 4.1 рџ”ґ Parent Fees вЂ” HARD CRASH (mismatch)
- **Client** (`shared/.../parent/domain/model/ParentFeatureModels.kt:8вЂ“30`): `FeeResponse{success, data: FeeData}` where `FeeData` **requires** `total_collected`, `collection_progress: Float`, `outstanding_fees`, `overdue_count`, `announcements: [FeeAnnouncementDto{id,title,time,description,open_rate,engagement,type}]`.
- **Server actually serving `/fees`** is `feature/parent/ParentFeesRouting.kt` (route at line 77): returns `call.ok(ParentFeesResponse{stats: FeesStats{total_collected, progress: Double, outstanding, overdue_count}, announcements:[FeesAnnouncement{id,title,time,desc,type}]})` (DTOs at lines 40вЂ“63).
- **Mismatch:** envelope `data` = `ParentFeesResponse`, so the client sees `data.stats.*` but its model demands `data.collection_progress`, `data.outstanding_fees`, and each announcement needs `description/open_rate/engagement` вЂ” server sends `desc` and omits `open_rate`/`engagement`. в†’ **`MissingFieldException` on deserialize.**
- **Trigger is immediate:** `FeeViewModel.kt` calls `loadFees()` from `init {}` (lines 41вЂ“42) the moment `ParentFeesScreenV2` mounts в†’ the Fees tab crashes on open.
- **Aggravating dead code:** `feature/user/ParentRouting.kt:70вЂ“86` defines a *correct* `FeeDataDto`/`FeeAnnouncementDto` that **would** match the client вЂ” but `ParentRouting` only registers `/scholarships` and `/announcements`, never `/fees`. So the correct DTO is unused and the wrong one wins. (Two files both `route("/api/v1/parent")`: `ParentRouting.kt:139` and `ParentFeesRouting.kt:76`.)

### 4.2 вњ… Parent Scholarships вЂ” parity OK, but DATA IS FAKE
- Server `ScholarshipDto/ScholarshipApplicationDto/ScholarshipsDataDto` (`ParentRouting.kt:91вЂ“116`) match the client (`ParentFeatureModels.kt:50вЂ“66`) field-for-field. Serialization is fine.
- **But the payload is hardcoded fiction** (`ParentRouting.kt:143вЂ“150`): `"Global Excellence STEM Award 2024" "$45,000"`, `"University of Applied Sciences вЂ” B.Arch Sustainable Urbanism"`, etc. No DB read. The Scholarships screen is a **static stub dressed as live data.**
- вњ… **RESOLVED (SCHOLAR, commit `774d655`)** вЂ” `/scholarships` now reads real rows from `ScholarshipsTable` (active, ordered by `position`) and `ScholarshipApplicationsTable` (scoped by JWT parent uid). Gamification fields are derived from real signals (`current_level` = highest child level; `profile_strength` scales with application count), and `DemoSeed.seedScholarships()` seeds demo rows idempotently. No more hardcoded fiction.

### 4.3 вњ… Parent Announcements вЂ” parity OK, real data
- Server `ParentAnnouncementDto` (`ParentRouting.kt:121вЂ“128`) == client (`ParentFeatureModels.kt:83вЂ“90`). Reads real rows from `AnnouncementsTable` scoped to the parent's children's schools (`ParentRouting.kt:171вЂ“199`). Good вЂ” this is one of the few genuinely-wired parent features.

### 4.4 вњ… Parent Track-Progress вЂ” parity OK, but content is a CMS template
- Server `TrackProgressDataDto` + nested `AchievementBadgeDto/AcademicCompetencyDto/PlayIndicatorDto` (`ParentRouting.kt:27вЂ“58`) == client `TrackProgressData` (`TrackProgressModels.kt:13вЂ“45`). Serialization fine.
- However `TrackProgressRouting.kt` populates competencies from a **CMS/`app_config` template**, not from per-child academic records вЂ” so two different children render identical "progress."

### 4.5 Teacher endpoints вЂ” wired both ways but unreachable
- Client `TeacherApi` calls `home/classes/attendance/marks/syllabus/homework/profile` (paths confirmed). Server mounts them (`TeacherRouting.kt:163`, `TeacherRoutingTasks.kt` `/attendance` 170, `/marks` 265, `/syllabus` 394, `/homework` 472) and they're registered (`Application.kt:183`). DTO parity not exhaustively diffed because the surface is **moot until teachers can be created (В§3.1)** вЂ” it can never be exercised in prod today.

## 13. REMEDIATION BLUEPRINT (design only вЂ” NO code changed in this audit)

> This is the plan for the **fixing phase** the user described (Supabase schema for mock data, one credential per profile type, schema-driven school search). Nothing here was applied in this audit.

1. **Single canonical schema doc.** Author one `docs/db/PROVISION.sql` (or a README) that runs, in order: `vidyasetu_schema.sql в†’ migration_001 в†’ migration_002 в†’ 02_teacher_schema.sql`. Fix the false header in `Tables.kt`. Add a boot-time `SchemaUtils` validation that logs any of the 36 tables missing and refuses to start in prod if so.
2. **Operational seed + one credential per profile type.** Extend `Seed.kt` (or add `DemoSeed.kt`) to insert: 1 `super_admin`, 1 `school_admin`, 1 `teacher` (which also fixes В§3.1 reachability), 1 `parent` + linked `child`, plus a demo `school`, `fee_records`, `announcements`, `exam_results`. Each with a known login. Move all the fabricated "Elena Rodriguez"-style analytics from `app_config` into proper tables so the frontend reads **Supabase data instead of hardcoded mocks**.
3. **Teacher provisioning path.** Add a school-admin endpoint to create a `teacher` `app_users` row (and the demo seed above), so teacher login becomes possible.
4. **Fix the fees crash.** Make `/fees` and the client agree on ONE contract. Simplest: delete `ParentFeesRouting`'s response shape and serve the already-correct `FeeDataDto` from `ParentRouting` (wire the dead DTO), OR change the client `FeeData` to match `ParentFeesResponse`. Then remove the duplicate `route("/api/v1/parent")` registration.
5. **Security.** Hard-fail boot if `JWT_SECRET` is unset/default in prod; switch passwords to bcrypt/argon2 with per-user salt; shorten access-token TTL and add server-side logout/revocation.
6. **Session + refresh.** Persist `userId`, `refreshToken`, `profileCompleted` in `PreferenceRepository`; add `AuthApi.refresh()`; `install(Auth)` + a 401 interceptor in the Koin HttpClient so expired tokens auto-refresh.
7. **Routing.** Map `super_admin` to a real admin destination in `EntryRole.from`; replace the profile-avatar `onLogout` with a real Profile screen (add a Profile tab or settings entry) and move logout there.
8. **Wire the orphans / mocks.** Call `/api/v1/parent/dashboard` from `ParentHomeScreenV2` (give it `ParentDashboardViewModel`, stop sharing `TrackProgressViewModel`); back `NotificationsScreenV2` and `ParentLinkChildScreenV2` with real endpoints; replace hardcoded scholarships with a DB-backed list.
