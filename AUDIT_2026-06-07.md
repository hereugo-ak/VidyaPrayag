## 1. 🔴 DATABASE / SUPABASE SCHEMA LAYER

### [DB-01] Two competing, incompatible "base" schemas in the repo
**Severity:** 🔴 CRITICAL
**Location:** `/supabase_schema` (root, v2.1) vs `docs/db/vidyasetu_schema.sql` vs `server/.../db/Tables.kt`
**Problem:** The repo ships **two different Postgres schemas that disagree on the core tables**:

| Concept | root `supabase_schema` (v2.1) | `docs/db/vidyasetu_schema.sql` (server's real target) |
|---|---|---|
| User table | `users` (PK = `auth.users(id)`, Supabase-Auth-bound, RLS) | `app_users` (standalone, phone-OTP, `password_hash`) |
| Parent↔child | `student_parent_link` + `students` (rich) | `children` (parent-owned, optional `student_code`) |
| Marks | `academic_records` (numeric, generated `percentage`) | `exam_results` (string score) + `assessments`/`assessment_marks` |
| Fees | `fee_structures` + `fee_payments` (full billing) | `fee_records` (flat line items, `USD` default) |
| Attendance | `daily_progress` (per-student, rich) | `attendance_records` (string status) |
| AI reports | `ai_reports` (PEWS, term reports) | *(none — no table, no route)* |
| OTP / sessions | *(none — uses Supabase Auth)* | `auth_otps`, `user_sessions`, `otp_delivery_attempts` |

**Evidence:**
- `server/.../db/Tables.kt:1-30` header claims it maps to `/supabase_schema`, but the actual table set (`app_users`, `auth_otps`, `children`, `fee_records`, `exam_results`, …) only matches `docs/db/vidyasetu_schema.sql` (`grep "create table"` lists `app_users`, `auth_otps`, `children`, `fee_records`, …).
- Root `supabase_schema` table list = `users, students, student_parent_link, academic_records, achievements, ai_reports, daily_progress, fee_structures, fee_payments, schools, school_directory` — **none of `app_users/children/fee_records/exam_results` exist there.**
- `DatabaseFactory.kt:9-16` says the two SQL files to run are `/supabase_schema` + `docs/backend/sql/01_supplementary_schema.sql`; but `migration_002` (line 22-24) says the run order is `vidyasetu_schema.sql → migration_001 → migration_002`. **The docs contradict each other on which base schema is canonical.**

**Impact:** If an operator runs the root `supabase_schema` (the file the user pointed at) the backend will fail at runtime — it queries `app_users`, which that schema never creates; and it never touches `users`/`student_parent_link`/`ai_reports`. The "Supabase schema in the repo" is **not the schema the app runs on.** This is the single most important structural defect.

### [DB-02] `schools.latitude` / `schools.longitude` missing from both base schemas
**Severity:** 🟠 HIGH
**Location:** `server/.../db/Tables.kt:184-185` (`SchoolsTable.latitude/longitude`) vs `supabase_schema:279-322` (no lat/long) vs `docs/db/vidyasetu_schema.sql:528-552` (no lat/long)
**Problem:** The server's `SchoolsTable` declares `latitude`/`longitude`, used by the parent "schools near me" discovery (Haversine in `ParentDashboardRouting.kt:97`). Neither base schema has these columns. They are added **only** by `docs/db/migration_002_segmentation_geo_assignments.sql:45-48`.
**Evidence:** `grep latitude supabase_schema` → 3 hits, **all in `school_directory`, none in `schools`**. `migration_002` line 13 explicitly documents the production crash: `column "latitude" of relation "schools" does not exist`.
**Impact:** On Postgres (where `AUTO_CREATE_TABLES` is off), forgetting `migration_002` makes **every** `SELECT … FROM schools` fail (Exposed selects all declared columns) → discovery, featured schools, onboarding all 500.

---

### [DB-03] Server `students` table is a thin mirror; root schema demands many NOT-NULL columns
**Severity:** 🟠 HIGH
**Location:** `Tables.kt:406-416` (`StudentsTable`: 9 cols) vs `supabase_schema:384-418` (`students`: ~30 cols, many `NOT NULL`: `roll_number`, `academic_year`, plus `UNIQUE(school_id,class_name,section,roll_number)`)
**Problem:** If the root v2.1 `students` table is the live one, any INSERT the backend attempts (it treats `students` as a read mirror, but other flows write `student_code`) will violate NOT-NULL/`academic_year` constraints the server never sets.
**Evidence:** Root `students` requires `academic_year TEXT NOT NULL` and `admission_number UNIQUE`; the server `StudentsTable` has neither.
**Impact:** Insert/seed of students against the v2.1 schema fails; against `vidyasetu_schema.sql` (thin) it works. Yet another reason the two schemas can't be mixed.

---

### [DB-04] No `ON DELETE` strategy alignment between code FKs and DB FKs
**Severity:** 🟡 MEDIUM
**Location:** `Tables.kt` (Exposed `.references()` mostly absent for child tables) vs `supabase_schema` (rich `ON DELETE CASCADE/RESTRICT/SET NULL`)
**Problem:** Server tables store FK-like UUIDs (`childId`, `schoolId`, `parentId`, `threadId`, `ptmEventId`, `homeworkId`, `assessmentId`) as **plain `uuid()` columns with no `.references()`**, so on SQLite there is no FK enforcement and on Postgres the constraint depends entirely on whichever SQL file was run. Deleting a school/child can orphan `fee_records`, `children`, `messages`, `assessment_marks`, etc.
**Evidence:** e.g. `Tables.kt:433 parentId = uuid("parent_id")` (no `.references`), `:573 threadId = uuid("thread_id")`, `:649 assessmentId = uuid("assessment_id")`.
**Impact:** Orphaned rows, referential drift, and queries that join on missing parents silently return nothing.

---

### [DB-05] RLS exists only in the root schema the app doesn't use
**Severity:** 🟠 HIGH
**Location:** `supabase_schema:842-1039` (full RLS) — but the server talks to `app_users`, not `users`/`auth.uid()`.
**Problem:** All the row-level-security policies are written against `auth.uid()` + `users` (Supabase Auth). The Ktor backend uses its own `app_users` + JWT and connects with the service/DB credentials, **bypassing RLS entirely.** So the carefully-written RLS is dead weight, and the real authorization is only the server's `requireSchoolContext()` checks.
**Evidence:** RLS `my_school_id()`/`my_role()` read `public.users WHERE id = auth.uid()`; server reads `app_users` (`SchoolAccess.kt:55-60`). The two never meet.
**Impact:** False sense of DB-level security. If anyone queries Supabase directly with the anon key expecting RLS to protect `app_users`/`children`/`fee_records`, **those tables have no policies at all** (they're not in the RLS section).

## 3. 🔴 AUTHENTICATION / SESSION / ROLE LAYER

### [AUTH-01] Teachers cannot be created → teacher login is unreachable
**Severity:** 🔴 CRITICAL
**Location:** `AuthRouting.kt:173-180` (`roleNormalised`), signup `:337-376`
**Problem:** Public signup runs `roleNormalised(req.role)` which can only yield `parent` or `school_admin` (and `super_admin` only if explicitly passed). There is **no route anywhere** that inserts an `app_users` row with `role = 'teacher'` (no school-admin "add teacher" endpoint, no invite flow).
**Evidence:** `grep "role.*=.*teacher|createTeacher|invite" server/src` → only comments/`PtmRouting` noise; no insert. `FacultyTable` rows are display-only and have **no `password_hash` and no login linkage**.
**Impact:** Login for a teacher requires an `app_users` row with `role='teacher'` + a password — which nothing can produce. The entire teacher vertical (8 screens, `api/v1/teacher/*`) is **dead on arrival** in production.

### [AUTH-02] `profile_completed`, `user_id`, `refresh_token` are never persisted
**Severity:** 🟠 HIGH
**Location:** `AuthRepositoryImpl.kt:73-79`
**Problem:** `saveSession()` writes **only** `setUserToken(response.token)` + `setUserRole(response.role)`. `profileCompleted`, `userId`, and `refreshToken` are dropped. `getSession()` returns the **in-memory** `cachedSession`, which is `null` after an app restart.
**Evidence:** `saveSession` body = 3 lines (token, role); `getSession(): AuthResponse? = cachedSession`.
**Impact:**
- Post-login gates (`NavGraphV2.kt:182-183`) read `getSession()?.profileCompleted ?: true` → after restart it's `null` → treated "completed" → **child-link / onboarding / first-login gates never fire for returning users.**
- No stored `refreshToken` → token refresh is impossible even if implemented.

### [AUTH-03] No global 401 handler / no token refresh in HTTP client
**Severity:** 🟠 HIGH
**Location:** `shared/.../di/Koin.kt:59-73` (HttpClient)
**Problem:** The `HttpClient` installs only `ContentNegotiation`, `HttpRedirect`, `HttpTimeout`. There is **no `install(Auth)`, no `bearer{}`, no `HttpResponseValidator`/401 interceptor**. Tokens are attached manually per call.
**Evidence:** `Koin.kt:59-73`; tokens added as `header("Authorization", "Bearer $token")` per API (e.g. `ParentApi.kt`).
**Impact:** When the access token expires, every protected call returns 401, repos surface it as a generic error, and screens show empty/error states with **no re-login prompt and no refresh** — the app appears to "break" after the JWT TTL with no recovery path.

### [AUTH-04] Profile-icon avatar is wired directly to logout (all 3 portals)
**Severity:** 🔴 CRITICAL
**Location:** `ParentPortalV2.kt:107,189-190`; `TeacherPortalV2.kt:113`; `SchoolPortalV2.kt:135`
**Problem:** The header avatar's `clickable { onExit() }` where `onExit = onLogout` → `MainViewModel.logout()` → `clearSession()` → back to landing. The Parent portal has **no profile tab at all** (`VNavItem` = Home/Academics/Fees/Activity only, `:94-97`), so the avatar is the only profile affordance and it logs you out.
**Evidence:** `ParentPortalV2.kt:189 Box(Modifier.clickable(...) { onExit() }) { VAvatar(name="Parent") }`, `:107 onExit = onLogout`.
**Impact:** Tapping your profile picture logs you out. Exactly the reported bug. Not a 401 (see AUTH-03), it's hard wiring.

### [AUTH-05] `super_admin` mis-routes into the Parent portal
**Severity:** 🟡 MEDIUM
**Location:** `NavGraphV2.kt:80-85` (`EntryRole.from`) + `:228-233` (`RolePortal`)
**Problem:** Server emits `super_admin` (`AuthRouting.kt:177`), but `EntryRole.from` only maps PARENT/ADMIN/SCHOOL_ADMIN/TEACHER → everything else (incl. `super_admin`) → `Unknown` → ParentPortal.
**Impact:** A super-admin lands in the parent UI with no admin surface.

### [AUTH-06] Logout never revokes the server session
**Severity:** 🟡 MEDIUM
**Location:** `AuthRepositoryImpl.kt:81-84`
**Problem:** `logout()` clears local prefs only; no call to revoke the `user_sessions` refresh token server-side. The refresh token remains valid until natural expiry.
**Impact:** Stolen/leaked refresh token survives "logout"; security/compliance gap.

### [AUTH-07] `MediaRouting` upload endpoint has no authentication
**Severity:** 🟠 HIGH
**Location:** `server/.../feature/media/MediaRouting.kt` (no `authenticate("jwt")` block — confirmed in the per-file auth scan)
**Problem:** Of all feature routers, `MediaRouting`, `AuthRouting`, `AppStatusRouting`, `VersionRouting`, `LandingRouting`, `SupportRouting` are NO-AUTH. The first four/last two are fine public; **media upload being public is not** — `POST /upload` can be hit by anyone.
**Evidence:** auth scan: `NO-AUTH media/MediaRouting.kt`; `post("/upload")` present.
**Impact:** Unauthenticated file upload into Supabase storage (cost/abuse/storage-poisoning risk).
