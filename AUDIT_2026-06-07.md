## 1. üî¥ DATABASE / SUPABASE SCHEMA LAYER

### 1.1 Two conflicting "source of truth" schemas
- **Root `/supabase_schema`** (Postgres v2.1) defines a *Supabase-native* model: `school_directory, schools, users (PK = auth.users.id, full RLS via auth.uid()/my_role()/my_school_id()), students (~30 cols), student_parent_link, daily_progress, academic_records, achievements, ai_reports, fee_structures, fee_payments`.
- **`docs/db/vidyasetu_schema.sql`** (CRLF line endings) defines the model the **server actually maps to**: `app_users, auth_otps, children, fee_records, exam_results, message_threads, messages, leave_requests, ptm_events, ptm_class_progress, users, schools (NO lat/long), students (thin 9-col)`.
- **The server (`server/.../db/Tables.kt`) maps to the SECOND schema, NOT the root one.** Yet `Tables.kt` lines 1‚Äì30 **falsely claim** it maps to `/supabase_schema` + `01_supplementary_schema.sql`. The header documentation is wrong.
- **Net:** `/supabase_schema` only creates **2 of the 36 server tables** (`schools`, `students`) and even those differ in column shape (root `students` ~30 cols vs server `StudentsTable` thin 9-col mirror at `Tables.kt:406`). Following the root schema alone leaves the server with 34 missing tables.

### 1.2 Definitive table-coverage matrix (the user's explicit "do we have all the DB tables?" ask)
`server/.../db/DatabaseFactory.kt` `allTables` (lines 44‚Äì83) registers **36 Exposed tables**. Postgres does NOT auto-create unless `AUTO_CREATE_TABLES=true`; otherwise you must run SQL by hand. There are two *documented* provisioning "recipes," and **both are incomplete**:

- **Recipe A** ‚Äî what `Tables.kt`/`DatabaseFactory.kt` instruct (`supabase_schema` + `docs/backend/sql/01_supplementary_schema.sql`):
  **MISSING 15 tables:** `assessments, assessment_marks, syllabus_units, homework, homework_submissions, teacher_periods, teacher_subject_assignments, children, fee_records, exam_results, message_threads, messages, leave_requests, ptm_events, ptm_class_progress`.
- **Recipe B** ‚Äî what `docs/db/migration_002_*.sql` run-order block (lines 22‚Äì24) instructs (`vidyasetu_schema.sql` ‚Üí `migration_001` ‚Üí `migration_002`):
  **MISSING 6 teacher tables:** `assessments, assessment_marks, syllabus_units, homework, homework_submissions, teacher_periods` (it omits `docs/backend/sql/02_teacher_schema.sql` entirely).
- **The ONLY complete recipe** is `vidyasetu_schema.sql` + `migration_001` + `migration_002` + `02_teacher_schema.sql` ‚Äî and **no single document lists all four together.** A new dev will guess one of the two incomplete recipes and get 500s on the teacher/messaging/ptm/fee endpoints.

### 1.3 Column drift that will bite at runtime
- `SchoolsTable.latitude`/`longitude` (`Tables.kt:184‚Äì185`) exist in the server model and are used by the Haversine distance sort in `ParentDashboardRouting.kt:97`, but they are **NOT** in `vidyasetu_schema.sql` (`schools` at ~line 528) nor in the root schema ‚Äî they are only added by `migration_002` (lines 45‚Äì48). Skip that migration ‚Üí `schools/discover` distance query throws "column does not exist."
- `FeeRecordsTable.currency` defaults to `"USD"` (`Tables.kt:464`) ‚Äî wrong default for an India-first product (should be `INR`).
- FK columns across `Tables.kt` are plain `uuid()` with **no `.references()`** ‚Üí no referential integrity; orphaned `children`/`fee_records` rows are possible and silent.

## 2. üî¥ SEED / DEMO-DATA LAYER ‚Äî why a fresh deploy is empty & unlogin-able

- The seeder also injects **fabricated analytics blobs** into `app_config` (see ¬ß7), so the only "data" present at boot is fake.

## 3. üî¥ AUTHENTICATION / SESSION / ROLE LAYER

### 3.1 Teachers can never be created (portal unreachable)
- `AuthRouting.kt` signup (`~348`) writes `app_users` and login (`~395‚Äì475`) reads role from DB. `roleNormalised` (lines 173‚Äì180) yields `parent / school_admin / teacher / super_admin`.
- **There is no code path anywhere that creates a `teacher` row.** No teacher self-signup, no school-admin "add teacher" endpoint that provisions an `app_users` record with role `teacher`. So `login` can never find a teacher ‚Üí the whole `/api/v1/teacher/*` surface (`TeacherRouting.kt`, `TeacherRoutingTasks.kt`) and the teacher portal screens are **dead in production**.

### 3.2 Forgeable JWT secret
- `JwtConfig.kt:40` ‚Äî `val secret by lazy { env("JWT_SECRET", "vidyaprayag-dev-secret-change-me") }`. If `JWT_SECRET` is unset in the Render env, the server signs with a **public, hardcoded string** ‚Üí anyone can forge an admin token. The header (lines 26‚Äì27) even admits it's "INSECURE." Must hard-fail at boot if unset in prod.
- Access-token TTL is **7 days** (`JwtConfig.kt:44`, `604800`s) and refresh is **30 days** (line 75) ‚Äî long-lived tokens magnify the forgery risk.

### 3.3 Unsalted password hashing
- `AuthRouting.kt:165` ‚Äî `hashPassword(p) = sha256Hex("pwd:$p")`. A single **static prefix, no per-user salt, no KDF (bcrypt/argon2)**. The whole password column is rainbow-table / GPU-crackable. Used at signup (line 348) and login compare (line 402).

### 3.4 Session persistence is lossy; client refresh is dead code
- Server **does** issue refresh tokens and persist `UserSessionsTable` (`AuthRouting.kt:362, 424, 504‚Äì506`) and exposes a working `/refresh` route validating `refresh_token_hash` (line 455).
- **But the client never uses any of it:**
  - `AuthApi.kt` exposes `checkUser/signup/login/sendOtp/verifyOtp/getUserDetails` ‚Äî **no `refresh()` method**. The server's refresh infrastructure is therefore **unreachable / dead code**.
  - `AuthRepositoryImpl.saveSession()` (lines 73‚Äì77) persists **only** `token` + `role` to prefs and stashes the full `AuthResponse` in an **in-memory `cachedSession`**. `getSession()` returns only `cachedSession` ‚Üí **`null` after app restart.**
  - `PreferenceRepository` (`core/prefs/PreferenceRepository.kt`) has fields for **only** `themeName`, `userRole`, `userToken` ‚Äî **no `profileCompleted`, no `userId`, no `refreshToken`.** These three are **permanently unrecoverable** after a restart, and the refresh token is never stored anywhere, so even adding a client `refresh()` wouldn't have a token to send.
- `Koin.kt` HttpClient (lines 59‚Äì73) installs **only** `ContentNegotiation`, `HttpRedirect`, `HttpTimeout`. **No `install(Auth)`, no 401 interceptor** ‚Üí an expired token surfaces as a generic error, never an auto-refresh.
- Silver lining (and a subtle inconsistency): most VMs read the token via `preferenceRepository.getUserToken()` (persisted), e.g. `FeeViewModel.kt:48`, so token-only calls survive restart ‚Äî but anything depending on `getSession()` (e.g. `profileCompleted` gating) breaks.

### 3.5 `super_admin` mis-routes to Parent portal
- `NavGraphV2.kt` `EntryRole.from` (lines 80‚Äì85) maps `PARENT / ADMIN / SCHOOL_ADMIN / TEACHER` but **not `super_admin`** ‚Üí it falls to `Unknown` ‚Üí routed to `ParentPortal` (line 231). A super-admin lands in a parent UI with no admin surface.
- Same file: `AuthedFlow` reads `getSession()?.profileCompleted ?: true` (lines 182‚Äì183) ‚Äî because `getSession()` is null after restart (¬ß3.4), this **defaults to `true`** and skips the profile-completion gate entirely on a cold start.

### 3.6 No server-side logout / revocation
- `AuthRepositoryImpl.logout()` clears prefs only ‚Äî it does **not** call the server to revoke the `UserSessionsTable` row. Refresh tokens remain valid server-side for 30 days after "logout."

## 4. üî¥ SERIALIZATION PARITY ‚Äî client DTO ‚Üî server DTO (per-endpoint)

Method: compared every `@SerialName`/field of the client `domain/model` DTO against the server `*Routing.kt` `@Serializable` DTO, plus the envelope (`ApiResponse{success, message, data}` from `core/ResponseExtensions.kt`).

### 4.1 üî¥ Parent Fees ‚Äî HARD CRASH (mismatch)
- **Client** (`shared/.../parent/domain/model/ParentFeatureModels.kt:8‚Äì30`): `FeeResponse{success, data: FeeData}` where `FeeData` **requires** `total_collected`, `collection_progress: Float`, `outstanding_fees`, `overdue_count`, `announcements: [FeeAnnouncementDto{id,title,time,description,open_rate,engagement,type}]`.
- **Server actually serving `/fees`** is `feature/parent/ParentFeesRouting.kt` (route at line 77): returns `call.ok(ParentFeesResponse{stats: FeesStats{total_collected, progress: Double, outstanding, overdue_count}, announcements:[FeesAnnouncement{id,title,time,desc,type}]})` (DTOs at lines 40‚Äì63).
- **Mismatch:** envelope `data` = `ParentFeesResponse`, so the client sees `data.stats.*` but its model demands `data.collection_progress`, `data.outstanding_fees`, and each announcement needs `description/open_rate/engagement` ‚Äî server sends `desc` and omits `open_rate`/`engagement`. ‚Üí **`MissingFieldException` on deserialize.**
- **Trigger is immediate:** `FeeViewModel.kt` calls `loadFees()` from `init {}` (lines 41‚Äì42) the moment `ParentFeesScreenV2` mounts ‚Üí the Fees tab crashes on open.
- **Aggravating dead code:** `feature/user/ParentRouting.kt:70‚Äì86` defines a *correct* `FeeDataDto`/`FeeAnnouncementDto` that **would** match the client ‚Äî but `ParentRouting` only registers `/scholarships` and `/announcements`, never `/fees`. So the correct DTO is unused and the wrong one wins. (Two files both `route("/api/v1/parent")`: `ParentRouting.kt:139` and `ParentFeesRouting.kt:76`.)

### 4.2 ‚úÖ Parent Scholarships ‚Äî parity OK, but DATA IS FAKE
- Server `ScholarshipDto/ScholarshipApplicationDto/ScholarshipsDataDto` (`ParentRouting.kt:91‚Äì116`) match the client (`ParentFeatureModels.kt:50‚Äì66`) field-for-field. Serialization is fine.
- **But the payload is hardcoded fiction** (`ParentRouting.kt:143‚Äì150`): `"Global Excellence STEM Award 2024" "$45,000"`, `"University of Applied Sciences ‚Äî B.Arch Sustainable Urbanism"`, etc. No DB read. The Scholarships screen is a **static stub dressed as live data.**
- ‚úÖ **RESOLVED (SCHOLAR, commit `774d655`)** ‚Äî `/scholarships` now reads real rows from `ScholarshipsTable` (active, ordered by `position`) and `ScholarshipApplicationsTable` (scoped by JWT parent uid). Gamification fields are derived from real signals (`current_level` = highest child level; `profile_strength` scales with application count), and `DemoSeed.seedScholarships()` seeds demo rows idempotently. No more hardcoded fiction.

### 4.3 ‚úÖ Parent Announcements ‚Äî parity OK, real data
- Server `ParentAnnouncementDto` (`ParentRouting.kt:121‚Äì128`) == client (`ParentFeatureModels.kt:83‚Äì90`). Reads real rows from `AnnouncementsTable` scoped to the parent's children's schools (`ParentRouting.kt:171‚Äì199`). Good ‚Äî this is one of the few genuinely-wired parent features.

### 4.4 ‚úÖ Parent Track-Progress ‚Äî parity OK, but content is a CMS template
- Server `TrackProgressDataDto` + nested `AchievementBadgeDto/AcademicCompetencyDto/PlayIndicatorDto` (`ParentRouting.kt:27‚Äì58`) == client `TrackProgressData` (`TrackProgressModels.kt:13‚Äì45`). Serialization fine.
- However `TrackProgressRouting.kt` populates competencies from a **CMS/`app_config` template**, not from per-child academic records ‚Äî so two different children render identical "progress."

### 4.5 Teacher endpoints ‚Äî wired both ways but unreachable
- Client `TeacherApi` calls `home/classes/attendance/marks/syllabus/homework/profile` (paths confirmed). Server mounts them (`TeacherRouting.kt:163`, `TeacherRoutingTasks.kt` `/attendance` 170, `/marks` 265, `/syllabus` 394, `/homework` 472) and they're registered (`Application.kt:183`). DTO parity not exhaustively diffed because the surface is **moot until teachers can be created (¬ß3.1)** ‚Äî it can never be exercised in prod today.

## 13. REMEDIATION BLUEPRINT (design only ‚Äî NO code changed in this audit)

> This is the plan for the **fixing phase** the user described (Supabase schema for mock data, one credential per profile type, schema-driven school search). Nothing here was applied in this audit.

1. **Single canonical schema doc.** Author one `docs/db/PROVISION.sql` (or a README) that runs, in order: `vidyasetu_schema.sql ‚Üí migration_001 ‚Üí migration_002 ‚Üí 02_teacher_schema.sql`. Fix the false header in `Tables.kt`. Add a boot-time `SchemaUtils` validation that logs any of the 36 tables missing and refuses to start in prod if so.
2. **Operational seed + one credential per profile type.** Extend `Seed.kt` (or add `DemoSeed.kt`) to insert: 1 `super_admin`, 1 `school_admin`, 1 `teacher` (which also fixes ¬ß3.1 reachability), 1 `parent` + linked `child`, plus a demo `school`, `fee_records`, `announcements`, `exam_results`. Each with a known login. Move all the fabricated "Elena Rodriguez"-style analytics from `app_config` into proper tables so the frontend reads **Supabase data instead of hardcoded mocks**.
3. **Teacher provisioning path.** Add a school-admin endpoint to create a `teacher` `app_users` row (and the demo seed above), so teacher login becomes possible.
4. **Fix the fees crash.** Make `/fees` and the client agree on ONE contract. Simplest: delete `ParentFeesRouting`'s response shape and serve the already-correct `FeeDataDto` from `ParentRouting` (wire the dead DTO), OR change the client `FeeData` to match `ParentFeesResponse`. Then remove the duplicate `route("/api/v1/parent")` registration.
5. **Security.** Hard-fail boot if `JWT_SECRET` is unset/default in prod; switch passwords to bcrypt/argon2 with per-user salt; shorten access-token TTL and add server-side logout/revocation.
6. **Session + refresh.** Persist `userId`, `refreshToken`, `profileCompleted` in `PreferenceRepository`; add `AuthApi.refresh()`; `install(Auth)` + a 401 interceptor in the Koin HttpClient so expired tokens auto-refresh.
7. **Routing.** Map `super_admin` to a real admin destination in `EntryRole.from`; replace the profile-avatar `onLogout` with a real Profile screen (add a Profile tab or settings entry) and move logout there.

## RA-01 üî¥ `students` table has no write path anywhere in the API
- **Where:** `server/.../db/Tables.kt` defines `StudentsTable`; grep across the whole
  server module for `StudentsTable.insert` / `StudentsTable.batchInsert` ‚Üí **zero hits.**
- **Read paths that depend on it:**
  `feature/parent/ParentLinkRouting.kt:144-145` resolves the canonical student by
  `(StudentsTable.schoolId eq schoolUuid) and (StudentsTable.rollNumber eq req.rollNumber)`;
  teacher dashboards count students via `TeacherRouting.studentCountFor(...)`.
- **Impact:** On a fresh Postgres/Supabase deployment the `students` table is *empty
  forever* ‚Äî no admin/onboarding/provisioning route ever inserts a row. Therefore:
  (a) parents **cannot** link a child (the `(school_id, roll_number)` lookup always
  misses), and (b) teacher "class strength" counts are always 0. The roster is a
  read-only island with no producer.
- **Why it matters for this task:** the seed **must** insert `students` directly
  (PART 3 ¬ß7) or every parent-link and teacher-roster verification (PART 4) fails.
- **Recommendation (code, out of scope here):** add a roster-import/admit route that
  writes `students`, or derive `students` from `children` on link. Tracked, not fixed.

## RA-02 üî¥ `is_active` is never enforced in any auth guard
- **Where:** `core/SchoolAccess.kt:56-93` (`resolveSchoolAndRole` ‚Üí `requireSchoolContext`)
  reads only `school_id` and `role` from `app_users`; it never selects or checks
  `is_active`. `core/TeacherAccess.kt:78` (`requireTeacherContext`) likewise.
- **Impact:** Deactivating a user (`is_active=false`) does **not** revoke access ‚Äî
  any still-valid JWT continues to authorize school/teacher/parent endpoints. The
  column is decorative. This is an auth edge case: "off-boarded" staff remain live
  until token expiry.
- **Recommendation:** add `AppUsersTable.isActive eq true` to the context resolvers.
  Out of scope (code change); seed therefore sets `is_active=true` on every account.

## RA-03 üî¥ Parent‚Üíchild link trusts client-supplied `school_id` + `roll_number`
- **Where:** `feature/parent/ParentLinkRouting.kt:62-63` deserialize `school_id` and
  `roll_number` straight from the request body; `:128-145` use them verbatim to look
  up a `students` row, then `:176-199` insert into `children`.
- **Impact (IDOR / enumeration):** a parent in School A can submit School B's
  `school_id` with a guessed `roll_number`. If a matching student exists, the parent
  links to **another school's child** ‚Äî there is no check that the requesting parent
  has any relationship to that school (parents legitimately have `school_id = NULL`,
  so there is nothing to compare against). Roll numbers are short and sequential ‚Üí
  trivially enumerable.
- **Mitigating factor:** requires knowing/guessing a valid `(school_id, roll_number)`
  pair, and child PII exposure is bounded by what the child endpoints return.
- **Recommendation:** gate linking behind an out-of-band verification (admit code,
  admin approval, or OTP to a school-registered guardian phone). Documented; the
  verify matrix (PART 4, TEST-ISO-04) explicitly attempts this cross-school link and
  asserts the **current** behavior so the human can confirm the risk locally.

## RA-04 üî¥ Announcement WhatsApp fan-out: no per-recipient tenant re-check
- **Where:** `feature/announcements/AnnouncementRouting.kt` ‚Äî `resolveRecipientPhones(...)`
  builds the recipient list for an announcement and the delivery dispatcher
  (`feature/auth/delivery/OtpDeliveryDispatcher.kt`) sends to each.
- **Impact (multi-tenancy):** the recipient query is the only tenant boundary; if the
  audience filter is mis-scoped (e.g. an audience_filter that matches phones across
  schools, or a future code path that passes a raw phone list), there is no second
  assertion that each resolved phone belongs to the announcement's `school_id`.
  A single off-by-one in the audience query leaks one school's message to another's
  parents over WhatsApp/SMS ‚Äî irreversible once sent.
- **Recommendation:** re-assert `recipient.school_id == announcement.school_id` (or,
  for parents, `child.school_id`) immediately before enqueueing each message.
  Documented; verify matrix TEST-NOTIF-02/03 sends a School-1 announcement and
  confirms no School-2 parent phone appears in the resolved set.

## RA-05 üî¥ `scholarships` / `scholarship_applications` have no SQL DDL
- **Where:** `db/Tables.kt` registers `ScholarshipsTable` + `ScholarshipApplicationsTable`
  (2 of the 38 Exposed tables). `db/DatabaseFactory.kt` `validateSchema()` refuses to
  boot on Postgres if any registered table is missing (unless `AUTO_CREATE_TABLES=true`).
  But `docs/db/PROVISION.sql` (the 4-file recipe) creates only **36** tables and its
  own header claims "all 36 tables" ‚Äî these two are absent. No migration creates them.
- **Impact (schema drift / boot failure):** a locked-down Supabase project provisioned
  strictly from `PROVISION.sql` is missing 2 tables ‚Üí `validateSchema()` hard-fails at
  boot, **and** the seed's scholarship `INSERT`s would error on a missing relation.
- **Disposition: FIXED in this task.** `scripts/schema-patch-2026-06-07.sql` adds both
  tables idempotently (PATCH-001, PATCH-002) using the exact Exposed column shapes,
  so the seed runs cleanly and the server boots. This is the one finding that *must*
  be remediated here because it gates everything downstream.

## RA-06 üî¥ OTP verify + parent signup are not one transaction
- **Where:** `feature/auth/OtpService.kt` (verify marks `auth_otps` consumed) and
  `feature/auth/AuthRouting.kt` (signup inserts `app_users` with
  `is_phone_verified=true`). These are separate DB operations, not wrapped together.
- **Impact (transaction boundary):** a crash/timeout between "OTP consumed" and
  "user row inserted" leaves the OTP burned but **no** account created ‚Äî the parent
  cannot retry with the same code and must request a new one; worst case a partial
  state if any intermediate write succeeded. Under concurrency two requests racing
  the same code could both pass the read before either marks it consumed.
- **Recommendation:** wrap consume+create in a single `transaction { }` with the OTP
  row `SELECT ‚Ä¶ FOR UPDATE`. Documented; verify matrix TEST-SIGNUP-03 covers replay.

## RA-07 üüÝ Attendance / marks upsert race
- **Where:** `feature/teacher/TeacherRoutingTasks.kt` attendance & marks handlers
  read an existing row then insert/update. `AttendanceRecordsTable` has a UNIQUE on
  `(school_id, date, type, person_id)` and `AssessmentMarksTable` similar.
- **Impact (race condition):** two teachers (or a double-tapped client) submitting the
  same `(school_id,date,person_id)` concurrently can both pass the existence read,
  then one `INSERT` wins and the other throws a unique-violation that isn't always
  caught as "treat as update." Result: a 500 instead of an idempotent upsert.
- **Recommendation:** use `INSERT ‚Ä¶ ON CONFLICT ‚Ä¶ DO UPDATE` at the DB layer (Exposed
  `upsert`). Documented; the **seed** sidesteps this by using `ON CONFLICT DO NOTHING`.

## RA-08 üüÝ Teacher provisioning creates the user but no assignment
- **Where:** `feature/school/TeacherProvisioningRouting.kt` inserts an `app_users` row
  (role `teacher`) but does not create any `teacher_subject_assignments`.
- **Impact (API contract):** `teacherAssignmentsFor(ctx)` (`core/TeacherAccess.kt:111-156`)
  scopes a teacher to rows matching `teacher_id` **or** `teacher_name`. A freshly
  provisioned teacher with no assignment row sees an empty dashboard until an admin
  separately calls the assignment route ‚Äî easy to miss, looks like a bug to the teacher.
- **Recommendation:** allow provisioning to accept initial assignments, or surface a
  "no classes yet" state. Documented; verify matrix TEST-PROV-01/02 walks this exact
  two-step and asserts the in-between empty state.

## RA-09 üüÝ Track-progress returns a CMS template indistinguishable from empty
- **Where:** `feature/parent/TrackProgressRouting.kt` returns a structured template
  (sections/labels) even when the linked child has no marks/attendance yet.
- **Impact (contract drift):** the client cannot tell "child exists but no data" from
  "fully populated template with zeros" ‚Äî both look like a real screen. Encourages the
  app to render confident-looking empty data.
- **Recommendation:** include an explicit `has_data: bool` / `data_completeness`
  field. Documented.

## RA-10 üüÝ Exam-results upsert silently overwrites grading history
- **Where:** `feature/school/ResultsRouting.kt` upserts `exam_results` keyed on
  `(school_id, student, subject, exam)`.
- **Impact (data-integrity):** re-submitting a corrected mark overwrites the prior
  value with no audit row ‚Äî there is no history table, so a grade dispute has no trail.
- **Recommendation:** append-only results + a `current` view, or an audit log.
  Documented.

## RA-11 üüÝ Fees `status` is a free-form string, not a CHECK enum
- **Where:** `db/Tables.kt` `FeeRecordsTable.status` is a plain varchar; the domain is
  `DUE` / `PAID` / `OVERDUE` (and parent fees routing branches on those literals).
- **Impact (schema drift):** a typo'd status (`Paid`, `paid`, `DUE `) silently slips
  in and the parent fees screen mis-buckets it. No DB guarantee.
- **Recommendation / disposition:** schema-patch ships an **optional, commented-out**
  `CHECK (status IN ('DUE','PAID','OVERDUE'))` (PATCH-003) the human can enable after
  confirming no legacy rows violate it. Not enabled by default (could break boot on
  dirty data).

## RA-12 üü° `announcements.event_id` UNIQUE is global, not per-school
- **Where:** `AnnouncementsTable.event_id` carries a global UNIQUE.
- **Impact (multi-tenancy):** two schools can never share an `event_id` value; for
  seeds this means na√Øve per-school identical IDs collide. Not a security leak, but a
  smell ‚Äî the natural key should be `(school_id, event_id)`.
- **Disposition:** the seed sidesteps it by prefixing every `event_id` with the school
  slug (`s1-‚Ä¶` / `s2-‚Ä¶`), so PART 3 stays collision-free without a schema change.

## RA-13 üü° CORS `anyHost()`
- **Where:** `server/.../Application.kt` installs CORS with `anyHost()`.
- **Impact (auth edge case):** any origin may call the API with credentials-less
  requests; acceptable for a public mobile-first API but worth pinning to known origins
  in prod to reduce CSRF-style surface on any future cookie auth. Documented.

## RA-14 üü° Parent dashboard ignores child `is_active`
- **Where:** `feature/parent/ParentDashboardRouting.kt` aggregates across all
  `children` of the parent without filtering inactive/archived children.
- **Impact (edge case):** a withdrawn child still contributes to dashboard rollups.
  Documented.

## RA-15 üü° `children.student_code` not FK-enforced to `students.student_code`
- **Where:** `db/Tables.kt` ‚Äî `ChildrenTable.studentCode` is a nullable string with no
  foreign key to `StudentsTable.studentCode`.
- **Impact (data-integrity):** a `children` row can reference a non-existent student
  code; drift between the two rosters is undetectable at the DB layer.
- **Disposition:** the seed keeps `children.student_code` exactly equal to the inserted
  `students.student_code` for every linked child (PART 3 ¬ß7/¬ß8).

## RA-16 üü° OTP hash has no pepper-version column
- **Where:** `feature/auth/OtpService.kt` stores `SHA-256(code:salt:pepper)` where
  pepper comes from env.
- **Impact (auth edge case):** rotating the pepper instantly invalidates **all**
  outstanding OTPs with no way to distinguish "wrong code" from "pepper rotated," and
  no `pepper_version` column to support gradual rotation. Documented.

## RA-17 üü° No per-phone OTP rate-limit / lockout counter
- **Where:** `auth_otps` table + `OtpService.kt` ‚Äî no `attempt_count` / `locked_until`.
- **Impact (auth edge case):** OTP codes can be brute-forced within their validity
  window with no server-side throttle beyond expiry. Documented.

## RA-18 üü° `updated_at` not maintained by a DB trigger
- **Where:** most tables in `db/Tables.kt` carry `updated_at` defaulted at insert; there
  is no `BEFORE UPDATE` trigger, so freshness depends on every handler remembering to
  set it.
- **Impact (data-integrity):** stale `updated_at` on any code path that forgets it.
  Documented.

## RA-19 üü° `school_id` NOT-NULL not uniformly guaranteed at DB layer
- **Where:** calendar/syllabus/related tables in `db/Tables.kt` ‚Äî most carry
  `school_id`, but the NOT-NULL + index discipline is not uniform across every
  tenant-scoped table.
- **Impact (multi-tenancy):** a NULL `school_id` row would be invisible to every
  tenant-scoped query yet still occupy the table ‚Äî orphaned, un-deletable via the API.
- **Disposition:** the seed always sets `school_id` explicitly on every row.

## Cross-cutting verification notes (deep pass #2)
- **Multi-tenancy model confirmed (three distinct scoping mechanisms):**
  - *School side* ‚Üí `app_users.school_id` via `requireSchoolContext`
    (`core/SchoolAccess.kt:78-93`).
  - *Teacher side* ‚Üí `teacher_subject_assignments` matched by `teacher_id` **or**
    `teacher_name` (`core/TeacherAccess.kt:111-156`).
  - *Parent side* ‚Üí `children.school_id` (parents themselves keep `school_id = NULL`;
    `feature/auth/AuthRouting.kt` signup path).
  The seed must satisfy **all three** independently ‚Äî an admin/teacher needs a real
  `school_id`, a teacher needs assignment rows, a parent needs `children` rows. This
  is why PART 3 seeds students + assignments + children, not just `app_users`.
- **Boot gate confirmed:** `db/DatabaseFactory.kt validateSchema()` enumerates 38
  tables and aborts on any missing one. With `scholarships`/`scholarship_applications`
  absent from `PROVISION.sql` (RA-05), the schema-patch is a **boot prerequisite**, not
  a nicety.
- **Password hashing confirmed reproducible outside the JVM:**
  `feature/auth/PasswordHasher.kt` = `PBKDF2WithHmacSHA256`, 120000 iterations,
  16-byte salt, 256-bit key, encoded `pbkdf2$<iter>$<b64salt>$<b64dk>`. Reproduced
  bit-for-bit in Python (`hashlib.pbkdf2_hmac('sha256', pw, salt, 120000, dklen=32)`)
  for all 9 staff accounts in the seed ‚Äî see PART 2 credentials file.
- **Parents need no password hash:** they authenticate via phone + OTP
  (`is_phone_verified=true`); the seed therefore creates parent `app_users` with a
  null/placeholder `password_hash` and relies on the OTP login path.
