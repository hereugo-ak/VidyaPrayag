## DOMAIN 1 ‚Äî AUTHENTICATION & SESSION

### 1.1 Session persistence ‚Äî what is written vs what is read (verify finding F)
- **Write** (`AuthRepositoryImpl.saveSession`, lines 80-84): `userToken`, `userRole`, `userId`, `refreshToken`, `profileCompleted`. ‚úÖ
- **Read on cold start** (`getSession`, lines 90-102): same 5 fields reconstructed from `PreferenceRepository`. ‚úÖ ‚Äî **except `name`, which is hard-set to `""`** (line 99). The display name is intentionally *not* persisted; each portal re-fetches it from the live token (SchoolHome‚Üí`/user/details`, TeacherHome‚Üí`/home`, ParentProfile‚Üíprofile API). This is acceptable: the name always comes from the **currently-logged-in user's** token, never a stale cache.
- **`PreferenceManager` (roomMain)** persists exactly those 5 keys + theme (lines 13-18). The written-set == read-set. ‚úÖ **Finding F verified closed.**

### 1.2 Role isolation at logout (verify ¬ß3 family)
- `clearSession()` (`PreferenceManager.kt:99-107`) removes **all 5** session keys (`USER_ROLE`, `USER_TOKEN`, `USER_ID`, `REFRESH_TOKEN`, `PROFILE_COMPLETED`). `AuthRepositoryImpl.logout()` also nulls the in-memory `cachedSession` (line 127). ‚úÖ **No session field from user A survives into user B's login.** When admin logs out and teacher logs in, the teacher's `/user/details` and `/home` calls use the *teacher's* fresh token ‚Üí teacher name + teacher school only. **Verified closed.**

### 1.3 RA-20 üî¥ Parent `/link-child` never flips `profile_completed`
- **Where:** `server/.../feature/parent/ParentLinkRouting.kt:121-212` (`POST /link-child`). The handler upserts the `ChildrenTable` row (lines 158-192) and returns the linked child ‚Äî but **never updates `AppUsersTable.profileCompleted`** for the parent `uid`. Grep confirms: `grep -n profileCompleted ParentLinkRouting.kt` ‚Üí **zero hits**.
- **Why it matters:** Parent signup sets `profileCompleted = false` (`AuthRouting.kt:366`). The client gate (`NavGraphV2.AuthedFlow`, lines 185-188) routes `EntryRole.Parent` to `ParentLinkChild` whenever `profileCompleted == false`. Because linking never flips the flag, **on the next login the server's `/login` returns `profile_completed=false` again** (`AuthRouting.kt:467`) ‚Üí the parent is dumped back into the link wizard forever.
- **Contrast:** the admin path *does* flip it (`OnboardingRouting.kt:568-571`, REVIEW final submit). Parent is the only role missing the flip.
- **Fix (TIER 0):** in the `Linked` branch of `/link-child`, `AppUsersTable.update({ id eq uid }) { it[profileCompleted] = true }` inside the same `dbQuery` transaction.

### 1.4 RA-21 üî¥ Logout is confirmation-less in all three portals
- **Where:**
  - Parent: `ParentProfileScreenV2.kt:132` ‚Äî `VButton("Log out", onClick = onLogout ‚Ä¶)` fires immediately.
  - Teacher: `TeacherProfileScreenV2.kt:118` ‚Äî same.
  - Admin: `SchoolSettingsScreenV2.kt:142` ‚Äî `SettingRow(‚Ä¶ "Account" ‚Ä¶ onClick = onLogout)` fires immediately.
- Grep for `confirm`/`dialog` near logout ‚Üí **zero hits**. There is no `AlertDialog`/confirmation anywhere in the logout path.
- **Why it matters:** LAW: "ZERO confirmation-less destructive actions (logout, delete, remove)."
- **Fix (TIER 0):** introduce a reusable confirm dialog and gate every `onLogout` behind it.

### 1.5 Parent auth flow routing (verify the DoD parent line)
- `ParentAuthScreenV2` pins role to `PARENT` (line 50), drives phone‚ÜíOTP via `AuthViewModel` (`onContinue`‚Üí`checkUser`‚Üí`sendOtp`; `onSubmit`‚Üí`login`/`signup`). On success ‚Üí `onAuthSuccess()` ‚Üí `NavGraphV2` recomposes into `AuthedFlow`.
- The flow is correct **once RA-20 is fixed**: a returning, already-linked parent has `profile_completed=true` ‚Üí routed straight to `ParentPortalV2`. The *only* reason a parent currently hits account setup on every login is RA-20.

### 1.6 Token scoping (verify)
- JWT carries only `sub`/`role`/`name` (`JwtConfig.kt:89-98`) ‚Äî **no `school_id` claim**. This is *intentional and safe*: every school query resolves `school_id` from the DB via `resolveSchoolAndRole()` (`SchoolAccess.kt:56-61`), so a forged/stale claim cannot widen access. ‚úÖ
- JWT secret hard-fails in prod if unset/default (`JwtConfig.kt:60-70`) ‚Äî finding E verified closed. Passwords use PBKDF2 via `PasswordHasher` with legacy self-heal (`AuthRouting.kt:177, 414-427`) ‚Äî ¬ß3.3 verified closed.

### 1.7 Server-side logout/revocation (verify ¬ß3.6)
- `POST /api/v1/auth/logout` (`AuthRouting.kt:518-545`) is JWT-guarded and revokes the matching (or all) `UserSessionsTable` rows. Client `AuthApi.logout()` (lines 89-97) calls it best-effort before clearing prefs (`AuthRepositoryImpl.logout` lines 119-129). ‚úÖ **Verified closed.**

## RA-27 üî¥ `is_active` is never enforced (login + guard)
- **Login** (`AuthRouting.kt` `/login`, lines ~408-471): after `lookupUserByIdentifier` the handler verifies password/OTP and issues a token ‚Äî it **never reads `AppUsersTable.isActive`**. Grep: `grep -n isActive AuthRouting.kt` ‚Üí the **only** hit is line 367, which *sets* `isActive=true` at signup. There is no `if (!row[isActive]) fail(...)` anywhere.
- **Guard** (`SecurityModule.kt:31-33`): the JWT `validate { }` block only checks the audience claim and wraps a `JWTPrincipal` ‚Äî it does **not** re-read the DB to confirm the user is still active. `requireSchoolContext()` (`SchoolAccess.kt:78-94`) reads `school_id`+`role` but **not** `is_active`.
- **Net effect:** Deactivating a user (`is_active=false`) does **nothing** ‚Äî they can still log in fresh, and any already-issued token (up to 1-day access / 30-day refresh) keeps working. There is no kill-switch for a compromised or off-boarded account. This is the most serious *new* finding.
- **Fix (later):** (a) reject login when `!row[isActive]`; (b) in the JWT `validate` block (or a thin per-request check in `requireSchoolContext`/`principalUserUuid`), reject when the user row is inactive; (c) optionally revoke that user's `UserSessionsTable` rows on deactivation.

## RA-28 üü° Web session is in-memory only
- **Where:** `shared/src/jsMain/.../InMemoryPreferenceManager.kt` and `wasmJsMain/.../InMemoryPreferenceManager.kt` back `PreferenceRepository` with `MutableStateFlow` only ‚Äî nothing touches `localStorage`/`IndexedDB`. `clearSession()` resets the in-memory flows.
- **Where it's bound:** `roomMain` (DataStore, persistent) is `dependsOn`'d only by `androidMain`, `jvmMain`, `iosArm64Main`, `iosSimulatorArm64Main` (`shared/build.gradle.kts:69-77`). **js and wasmJs are NOT** ‚Üí they fall back to the in-memory manager.
- **Net effect:** On Android + iOS (the shipping mobile targets) session persists correctly. On **web**, a page reload drops token/role/userId/refreshToken ‚Üí the user is bounced to the landing funnel. Acceptable only if web is dev-only; if web is a launch target this is a üî¥. Filed üü° pending confirmation of web's launch status.

## RA-29 üü° Double Authorization header + cache staleness on refresh
- **Double header:** the global `HttpClient` installs `Auth { bearer { loadTokens {‚Ä¶} } }` (`Koin.kt:88-94`) which attaches `Authorization: Bearer <access>` to **every** request. Yet most API clients *also* set the header manually ‚Äî e.g. `AuthApi.getUserDetails` (`AuthApi.kt:69`), every `ParentApi` method (`header("Authorization","Bearer $token")`), `AnnouncementsApi`, etc. The two paths can disagree (manual token from a VM-read pref vs the plugin's `loadTokens` read), and the plugin's `sendWithoutRequest`/precedence is implicit. Result is fragile and makes the auto-refresh path partially moot for clients that hardcode the header.
- **Cache staleness:** the plugin's `refreshTokens` block writes the new access/refresh to `prefs` (`Koin.kt:117-118`) but does **not** update `AuthRepositoryImpl.cachedSession`. Any code path that later calls `getSession()` returns the **stale in-memory** token until the next cold start, even though `prefs` are fresh.
- **Net effect:** Works today (token-bearing calls succeed) but is double-wired and inconsistent; a future change to either path can silently send the wrong/old token. **Fix:** pick ONE source of truth ‚Äî let the `Auth` plugin own the header and strip the manual `header("Authorization", ‚Ä¶)` from every API client; and have `refreshTokens` also refresh `cachedSession` (or drop the in-memory cache entirely and always read prefs).

## RA-30 üü° Fragile server-error message extraction
- **Where:** `NetworkResult.kt:86-91`. On a non-2xx, the message is pulled via `errorBody.substringAfter("\"message\":\"").substringBefore("\"")`. This breaks when: the message contains an escaped quote (`\"`), the JSON is pretty-printed with spaces (`"message" : "‚Ä¶"`), or `message` is null. The user then sees a truncated/blank error instead of the server's real reason (e.g. "Too many OTP requests").
- **Fix:** deserialize the standard `{success,message,data}` envelope with `kotlinx.serialization` (lenient) and read `.message`, falling back to the status line only on parse failure.
- **‚úÖ RESOLVED** `203875e` ‚Äî added `extractErrorMessage(errorBody, statusCode)` (lenient `ErrorEnvelope` deserialize ‚Üí `message` then `error` ‚Üí trimmed body ‚Üí `"Server returned <status>"`); `safeApiCall` now calls it instead of `substringAfter`.

## RA-31 üüÝ Parent dashboard shows only the first child
- **Where:** `ParentDashboardRouting.kt:151-155` ‚Äî `ChildrenTable ‚Ä¶ where parentId == uid AND isActive ‚Ä¶ orderBy createdAt ASC ‚Ä¶ .firstOrNull()`. Only the earliest-linked active child is returned; `child_summary` is a single object, and there is **no child-list / child-switcher** anywhere in the parent UI.
- **Net effect:** A parent with two or more children at the same (or different) school sees exactly one. The others' progress/attendance/fees are unreachable from the dashboard. `is_active` *is* respected here (so RA-14's is_active worry is closed), but **multi-child** is a genuine gap.
- **Fix:** return a `children: []` array (and let the client render a selector), or at minimum document single-child as an intentional MVP limit.

## RA-32 üü° No teacher-credential recovery
- **Where:** `TeacherProvisioningRouting.kt` returns the new teacher's identifier in the `POST` response (lines 121-131) but there is **no** endpoint to (a) re-display/download the initial credential later, nor (b) reset/resend a teacher's password or trigger a teacher OTP from the admin side.
- **Net effect:** Once the admin dismisses the create-teacher result, the only recovery is a manual DB edit. The Settings copy literally promises "download credentials" (`SchoolSettingsScreenV2.kt:138`).
- **Fix:** add an admin-scoped `POST /api/v1/school/teachers/{id}/reset-password` (re-issues an initial password, school-scoped) and surface the credential in the add-teacher flow added by RA-22.

## RA-33 üî¥ `OTP_DEV_RETURN_CODE` defaults to `"true"` ‚Üí OTP echoed in prod
- **Where:** `OtpService.kt:125-127` ‚Äî `env("OTP_DEV_RETURN_CODE", "true").equals("true", true)`. The **default is `"true"`**, so unless DevOps explicitly sets `OTP_DEV_RETURN_CODE=false`, the value flows into `OtpSendResult.Sent.devCode` ‚Üí serialized as `dev_code` in `SendOtpResponse` (AuthRouting:89, 251).
- **Net effect:** Any anonymous caller hits public `POST /api/v1/auth/send-otp { identifier }` and **reads the OTP straight back in the JSON**, then logs in / signs up as that phone number. This is a full pre-auth account-takeover primitive if the env isn't overridden.
- **Fix (later):** flip the default to `"false"` and tie "echo allowed" to `!isProduction` (mirror `JwtConfig.isProduction` = `DATABASE_URL` present). In prod the field must always be `null`.

## RA-34 üî¥ `is_active` not enforced at login / refresh / school guard
- **Where:** written at `AuthRouting.kt:367` (`it[isActive] = true`); **read nowhere** in `feature/auth/**` or `core/**` (grep confirms only TeacherAccess uses `isActive`, and that's the *assignment* table, not the user). `/login` (408-471), `/refresh` (474-512), and `requireSchoolContext()` (SchoolAccess:78-94) all ignore it.
- **Net effect:** Deactivating/expelling a user has **no effect** ‚Äî they keep their access token, can **mint fresh tokens via `/refresh`**, and pass every school guard. Supersedes/extends RA-27 by adding the `/refresh` and guard entry points.
- **Fix (later):** in `/login` reject inactive before issuing; in `/refresh` re-load the user and reject if `!isActive`; in `requireSchoolContext()` (and a parallel parent/teacher guard) treat `!isActive` as 403. Optionally re-check in JWT `validate{}` for instant revocation.

## RA-35 üüÝ Refresh tokens are never rotated
- **Where:** `AuthRouting.kt:497-505` ‚Äî on refresh, only `lastUsedAt` is updated and `req.refreshToken` is returned **unchanged**.
- **Net effect:** No rotation ‚áí no reuse-detection. A stolen refresh token is valid for the entire 30-day window with the legitimate user none the wiser (both can use it indefinitely).
- **Fix (later):** issue a NEW refresh token on every `/refresh`, revoke the old session row, and persist the new hash. Detect reuse of a revoked refresh token as a theft signal ‚Üí revoke the whole family.

## RA-36 üüÝ No request-body size limit on JSON endpoints
- **Where:** `Application.kt` installs no body/size limit; only `MediaRouting.kt:73` caps bytes (multipart, 25 MB). All other `call.receive<‚Ä¶>()` buffer unbounded.
- **Net effect:** A single large POST (e.g. to public `/login` or `/signup`) can OOM the dyno ‚Üí DoS.
- **Fix (later):** install a global max-content-length guard (Ktor `RequestValidation`/plugin or a `intercept(ApplicationCallPipeline.Plugins)` that rejects `Content-Length` over N), e.g. 1 MB for JSON routes.

## RA-37 üüÝ CORS `anyHost()` + `Authorization` in all envs
- **Where:** `Application.kt:109-122`.
- **Net effect:** Open CORS is fine for dev but in prod any origin can drive authenticated requests with a captured bearer token.
- **Fix (later):** gate `anyHost()` behind `!isProduction`; in prod allow only the known web/app origins via `allowHost(...)`.

## RA-38 üü° OTP send/verify are multi-statement, not transactional (TOCTOU)
- **Where:** `OtpService.send()` resend-limit check (156-191) and `verify()` attempt-count update (337-365) each span **separate** `dbQuery{}` calls.
- **Net effect:** Concurrent requests interleave between the read and the update, so `maxResendsPerHour`/`maxAttempts` can be exceeded ‚Äî the brute-force lock is weaker than advertised.
- **Fix (later):** wrap each check-then-update in a single `dbQuery {}` transaction (one `transaction` block) and/or use a `SELECT ‚Ä¶ FOR UPDATE` row lock on the `auth_otps` row.

## RA-39 üü° `role` claim never enforced; staff == admin on school routes
- **Where:** `SecurityModule.kt:49` defines `principalRole()` ‚Äî **0 authorization call sites** (grep). Authz is role-from-DB in `requireSchoolContext()` (SchoolAccess:85, `SCHOOL_ROLES = {school_admin, school_staff, admin}`).
- **Net effect:** Reading role from DB is the *correct* call (a forged/stale claim can't widen access ‚Äî keep it). **But** there's no admin-vs-staff granularity: any `school_staff` row can provision teachers, delete media, send school-wide announcements, etc.
- **Fix (later):** introduce `requireSchoolAdmin()` (role ‚àà {school_admin, admin}) for privileged writes (teacher provisioning, media delete, announcement create) and keep `requireSchoolContext()` for reads/shared writes.

## RA-40 üü¢ `DEBUG_ERRORS` leak + stderr-only logging
- **Where:** `ErrorHandling.kt:57-62`. Default is safe ("Something went wrong"); flipping `DEBUG_ERRORS=true` returns `cause.message` to clients.
- **Fix (later):** keep default; document that `DEBUG_ERRORS` must never be set in prod; route `printStackTrace()` through SLF4J/Sentry.

## RA-41 üü° User enumeration + no password-login throttle
- **Where:** `/login` distinguishes `USER_NOT_FOUND` (409) from `INVALID_CREDENTIALS` (415); password path has no rate limit (only OTP path does, via OtpService).
- **Net effect:** Account enumeration + unthrottled credential stuffing on the email/password path.
- **Fix (later):** return a single generic "Invalid credentials" for both branches; add a per-IP / per-identifier login attempt throttle (reuse the OTP rate-limit pattern or a simple sliding-window table).
