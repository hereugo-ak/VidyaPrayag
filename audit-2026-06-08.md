## DOMAIN 1 — AUTHENTICATION & SESSION

### 1.1 Session persistence — what is written vs what is read (verify finding F)
- **Write** (`AuthRepositoryImpl.saveSession`, lines 80-84): `userToken`, `userRole`, `userId`, `refreshToken`, `profileCompleted`. ✅
- **Read on cold start** (`getSession`, lines 90-102): same 5 fields reconstructed from `PreferenceRepository`. ✅ — **except `name`, which is hard-set to `""`** (line 99). The display name is intentionally *not* persisted; each portal re-fetches it from the live token (SchoolHome→`/user/details`, TeacherHome→`/home`, ParentProfile→profile API). This is acceptable: the name always comes from the **currently-logged-in user's** token, never a stale cache.
- **`PreferenceManager` (roomMain)** persists exactly those 5 keys + theme (lines 13-18). The written-set == read-set. ✅ **Finding F verified closed.**

### 1.2 Role isolation at logout (verify §3 family)
- `clearSession()` (`PreferenceManager.kt:99-107`) removes **all 5** session keys (`USER_ROLE`, `USER_TOKEN`, `USER_ID`, `REFRESH_TOKEN`, `PROFILE_COMPLETED`). `AuthRepositoryImpl.logout()` also nulls the in-memory `cachedSession` (line 127). ✅ **No session field from user A survives into user B's login.** When admin logs out and teacher logs in, the teacher's `/user/details` and `/home` calls use the *teacher's* fresh token → teacher name + teacher school only. **Verified closed.**

### 1.3 RA-20 🔴 Parent `/link-child` never flips `profile_completed`
- **Where:** `server/.../feature/parent/ParentLinkRouting.kt:121-212` (`POST /link-child`). The handler upserts the `ChildrenTable` row (lines 158-192) and returns the linked child — but **never updates `AppUsersTable.profileCompleted`** for the parent `uid`. Grep confirms: `grep -n profileCompleted ParentLinkRouting.kt` → **zero hits**.
- **Why it matters:** Parent signup sets `profileCompleted = false` (`AuthRouting.kt:366`). The client gate (`NavGraphV2.AuthedFlow`, lines 185-188) routes `EntryRole.Parent` to `ParentLinkChild` whenever `profileCompleted == false`. Because linking never flips the flag, **on the next login the server's `/login` returns `profile_completed=false` again** (`AuthRouting.kt:467`) → the parent is dumped back into the link wizard forever.
- **Contrast:** the admin path *does* flip it (`OnboardingRouting.kt:568-571`, REVIEW final submit). Parent is the only role missing the flip.
- **Fix (TIER 0):** in the `Linked` branch of `/link-child`, `AppUsersTable.update({ id eq uid }) { it[profileCompleted] = true }` inside the same `dbQuery` transaction.

### 1.4 RA-21 🔴 Logout is confirmation-less in all three portals
- **Where:**
  - Parent: `ParentProfileScreenV2.kt:132` — `VButton("Log out", onClick = onLogout …)` fires immediately.
  - Teacher: `TeacherProfileScreenV2.kt:118` — same.
  - Admin: `SchoolSettingsScreenV2.kt:142` — `SettingRow(… "Account" … onClick = onLogout)` fires immediately.
- Grep for `confirm`/`dialog` near logout → **zero hits**. There is no `AlertDialog`/confirmation anywhere in the logout path.
- **Why it matters:** LAW: "ZERO confirmation-less destructive actions (logout, delete, remove)."
- **Fix (TIER 0):** introduce a reusable confirm dialog and gate every `onLogout` behind it.

### 1.5 Parent auth flow routing (verify the DoD parent line)
- `ParentAuthScreenV2` pins role to `PARENT` (line 50), drives phone→OTP via `AuthViewModel` (`onContinue`→`checkUser`→`sendOtp`; `onSubmit`→`login`/`signup`). On success → `onAuthSuccess()` → `NavGraphV2` recomposes into `AuthedFlow`.
- The flow is correct **once RA-20 is fixed**: a returning, already-linked parent has `profile_completed=true` → routed straight to `ParentPortalV2`. The *only* reason a parent currently hits account setup on every login is RA-20.

### 1.6 Token scoping (verify)
- JWT carries only `sub`/`role`/`name` (`JwtConfig.kt:89-98`) — **no `school_id` claim**. This is *intentional and safe*: every school query resolves `school_id` from the DB via `resolveSchoolAndRole()` (`SchoolAccess.kt:56-61`), so a forged/stale claim cannot widen access. ✅
- JWT secret hard-fails in prod if unset/default (`JwtConfig.kt:60-70`) — finding E verified closed. Passwords use PBKDF2 via `PasswordHasher` with legacy self-heal (`AuthRouting.kt:177, 414-427`) — §3.3 verified closed.

### 1.7 Server-side logout/revocation (verify §3.6)
- `POST /api/v1/auth/logout` (`AuthRouting.kt:518-545`) is JWT-guarded and revokes the matching (or all) `UserSessionsTable` rows. Client `AuthApi.logout()` (lines 89-97) calls it best-effort before clearing prefs (`AuthRepositoryImpl.logout` lines 119-129). ✅ **Verified closed.**

## DOMAIN 3 — MULTI-TENANCY & DATA ISOLATION

- **Core guard is correct.** `requireSchoolContext()` (`SchoolAccess.kt:78-94`) resolves `school_id` + `role` from `app_users` by JWT `sub`, never from the request body/query. Every school route audited (`teacherProvisioningRouting`, `announcementRouting`, `schoolAnalyticsRouting`, `onboardingRouting`) begins with `val ctx = call.requireSchoolContext() ?: return@…`.
- **WhatsApp fan-out (RA-04) verified closed:** `sync-whatsapp` constrains every announcement id to `ctx.schoolId`, rejects foreign ids with 403, and `resolveRecipientPhones()` filters parents/children by `schoolId` at every branch (`AnnouncementRouting.kt:383-446`).
- **Parent fees (`/fees`)** scoped by `parentId eq uid` — no cross-parent leak (`ParentFeesRouting.kt:89`).
- **`/link-child` (RA-03)**: resolves the canonical student by `(schoolId, rollNumber, isActive)` (`ParentLinkRouting.kt:142-148`); a parent cannot link a child from another school by id alone, but **roll-number enumeration within a chosen school is still possible** (no rate limit). Carried as 🟡 (RA-03, documented previously). No data leak — only existence-probing.
- **No CRITICAL cross-school read leak found in this pass.** RA-22's missing DELETE route, when added, MUST scope by `ctx.schoolId` to avoid introducing an IDOR.
